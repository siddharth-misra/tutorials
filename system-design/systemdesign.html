Formula: Availability = MTTF / (MTTF + MTTR)

This means availability is the fraction of total time the system is up: MTTF (Mean Time To Failure, the average healthy runtime before the next outage) divided by MTTF + MTTR, where MTTR (Mean Time To Recovery, the average time to restore service after an outage) is the downtime portion. Higher MTTF and lower MTTR increase availability.

Common Capacity Estimation Steps

  1. Daily active users × actions per day = daily events.
  2. Daily events / 86,400 = average RPS; multiply by 5–10× for peak.
  3. Average payload size × RPS = bandwidth.
  4. Storage = payload × events/day × retention days.
  5. Shards = total storage / per-node capacity.
  6. Sanity-check latency: sum up each hop.

Multi-Region: Active-Active vs Active-Passive

Model What it means Trade-offs
Active-Passive One region serves all traffic; the second is on standby. Failover is triggered manually or automatically on primary failure. Simpler to reason about. Failover takes time (RTO minutes–hours). The passive region is mostly idle cost. Write consistency is easier — one region owns writes.
Active-Active Both regions simultaneously serve traffic and accept writes. Traffic is routed by geography (nearest region). Near-zero RTO for regional failure. Lower latency for global users. Requires conflict resolution for concurrent writes to the same record. Much harder to make strongly consistent.

When to choose each:

  • Active-Passive: financial systems, single-tenant SaaS, teams without multi-region ops experience.
  • Active-Active: global consumer apps (social, e-commerce), where per-region latency matters and eventual consistency is tolerable (DynamoDB Global Tables, Cassandra multi-DC writes, CockroachDB).

Key concepts that appear with multi-region:

  • RTO (Recovery Time Objective): how quickly service must be restored. Active-Active targets seconds; Active-Passive typically targets minutes.
  • RPO (Recovery Point Objective): how much data loss is acceptable. Synchronous replication gives RPO ≈ 0; asynchronous replication accepts RPO of seconds to minutes.
  • Write conflict resolution: last-write-wins (LWW), vector clocks, CRDTs, or application-level merge logic.
  • Quorum writes across regions: strong consistency is possible but adds cross-region latency (~100 ms) to every write.

Chapter 1: System Design Interview Framework

The Interview Loop

Use the same structure for nearly every FAANG-style system design question:

  1. Clarify requirements and constraints.
  2. Estimate scale and capacity.
  3. Define the API contract.
  4. Draw the high-level design.
  5. Deep dive on bottlenecks and failure modes.
  6. Discuss trade-offs and evolution over time.

What To Ask First

  • Who are the users?
  • What is the read/write ratio?
  • What latency (how long one request or operation takes) and availability (how often the system stays usable) targets matter?
  • What is in scope vs. out of scope?
  • Is correctness, freshness, or cost the primary constraint?

How To Present The Answer

  • Start broad, then narrow to the hard parts.
  • Name one or two explicit assumptions.
  • Keep the request path and data path separate.
  • Call out the single biggest bottleneck early.
  • Finish with a clear trade-off and a future evolution path.

What Interview-Ready Depth Actually Means

A strong answer is not just a checklist. It should work at three layers at once:

  1. Product layer — what user or business problem the system solves.
  2. Mechanism layer — how the request actually flows through caches, services, queues, and storage.
  3. Operations layer — what breaks in production, how you detect it, and how you recover safely.

This is the difference between saying "use Redis" and actually explaining the design. A deep explanation names the cache key shape, TTL (Time To Live, meaning how long cached or temporary data stays valid) policy, invalidation trigger, fallback behavior on a miss, and failure mode when Redis is down. If you say "use Kafka," you should be able to explain partitioning, ordering, retry behavior, and how consumers avoid duplicate side effects. If you say "use Postgres," you should be able to explain why transactions (groups of operations that succeed or fail together) matter, which indexes (extra data structures that make lookups faster) exist, and what happens when replicas (extra copies of data on other machines) lag.

In practice, most strong system-design discussions move repeatedly between these layers. A notification service is not only "API -> queue -> workers." The real design also includes delivery semantics, idempotency (making duplicate processing safe), provider outage behavior, queue backlog growth, and how cost controls interact with retries. That is the level of detail this document should target.

Capacity Estimation Template

  1. Estimate QPS or daily event volume.
  2. Estimate object size and total storage.
  3. Estimate bandwidth on ingress and egress.
  4. Convert storage into shard/server count.
  5. Check whether the result fits the latency budget from Chapter 3.

Failure Analysis Template

  • Single point of failure
  • Hot partition or hot key (popular keys that receive far more traffic than average)
  • Backpressure (slowing incoming work when downstream systems are overloaded) and queue growth
  • Data loss vs. data duplication
  • Slow downstream dependency

Real-world example:

Interview loops at Google, Meta, and Amazon typically expect this exact progression: requirements, scale, API, architecture, bottlenecks, then trade-offs.

What Good API Design Looks Like in an Interview

In a real interview, the API step is where you show that you have thought about the interface before drawing boxes. For a notification system, a minimal but well-formed answer looks like:

http
POST /v1/notifications          # Trigger a new notification
GET  /v1/notifications/{id}     # Check delivery status

Request body:

json
{
  "recipient_id": "user_456",
  "channels": ["push", "email"],
  "event_type": "ride_cancelled",
  "payload": { "ride_id": "r_789", "reason": "driver_unavailable" }
}

What the interviewer is listening for: idempotency key on POST (so retries are safe), versioning (/v1/), and whether channels are explicit or implicit. This same pattern applies to any system: define the smallest API that satisfies the requirements, then explain what you left out and why.

Interview Trade-Off Questions

  1. If the interviewer gives you only 20 minutes, which steps do you compress and which ones do you never skip?
  2. When should you push back on a requirement because it makes the design unrealistic, instead of accepting it and moving on?
  3. If you have time for only one deep dive, how do you choose between scaling, data model, and failure handling?

Chapter 2: Networking & Internet Fundamentals

Problem Statement

Understand how requests move from a client to a service and back, and how the networking stack affects latency (how long one request or operation takes), reliability, and scale.

This chapter ties together the pieces that show up throughout the book: HTTP (the protocol browsers and APIs use for requests and responses)/HTTPS (HTTP protected with TLS encryption), REST APIs (HTTP APIs organized around resources and standard verbs), gRPC (a binary RPC framework for fast service-to-service calls), WebSockets (long-lived two-way connections between client and server), TCP (a reliable transport protocol for moving data between machines)/IP, DNS (the internet phonebook that maps names to IP addresses), load balancing, reverse proxies (front-end servers that forward requests to backend services), CDNs, SSL (the older name often used for TLS network encryption)/TLS (the encryption layer that protects data while it travels over the network), and API gateways (single entry points that route, protect, and observe APIs).

Plain-English Walkthrough

When a user taps "Place Order" in a mobile app, the phone is not talking directly to your business logic. It first resolves DNS, opens a TCP connection, negotiates TLS, reaches edge infrastructure such as a CDN (content delivery network, edge servers that cache content near users) or API gateway, and only then arrives at the application servers. If any of those hops is slow or broken, the user experiences "the app is slow" even if the database and application code are healthy.

A practical way to study networking is to trace one request end to end: name lookup, connection setup, routing, authentication, application work, and response delivery. That mental model turns a long list of protocols into one understandable request path.

Real-world example:

When a user opens app.example.com, DNS resolves the hostname, Cloudflare may terminate TLS at the edge, an API gateway routes the request, Nginx or HAProxy forwards it to an application server, and the server responds over HTTP or gRPC. If the request is for a product image or JavaScript bundle, the CDN may serve it entirely from edge cache. If it is for /checkout, the request continues to the origin because it needs fresh application logic and database reads.

Detailed Request Lifecycle

An end-to-end request is usually more layered than an interview whiteboard diagram suggests.

  1. Local caches first — the browser or mobile client may already have DNS answers, cached assets, open TCP connections, TLS session tickets, or even a service-worker response.
  2. DNS resolution — the client checks local and recursive-resolver caches before contacting authoritative DNS. TTL (Time To Live, meaning how long cached or temporary data stays valid) choice is a real trade-off: shorter TTLs improve failover (switching traffic or leadership after a failure) agility, but reduce cache efficiency.
  3. Connection setup — TCP performs a three-way handshake, then TLS negotiates encryption and server identity. HTTP/2 (a faster version of HTTP that can multiplex many requests on one connection) and HTTP/3 (the newer HTTP version that runs over QUIC for faster recovery from packet loss) exist partly to amortize this setup cost across many requests.
  4. Edge processing — CDN, WAF, DDoS protection, and API gateway rules may run before the origin is touched. This is where coarse rate limiting (controlling how fast a client may send requests), bot filtering, TLS termination, and caching decisions often happen.
  5. Application routing — reverse proxies and load balancers (front doors that spread traffic across multiple servers) decide which service instance gets the request. Inside the fleet, service discovery (the process of finding where a service is currently running) or a mesh determines the next hop.
  6. Business logic and storage — only here does the application execute auth checks, cache lookups, database queries, queue publishes, and downstream RPC calls.
  7. Response shaping — the response may be compressed, chunked, cached, signed, or streamed. Cache-Control headers determine whether future requests stop at the edge instead of reaching origin.

This matters because latency is additive. A 20 ms DNS miss, a 25 ms TLS handshake, a 15 ms gateway hop, a 30 ms database query, and a 40 ms downstream RPC already create a user-visible delay even though no single component looks catastrophic. Most slow systems are not broken by one thing; they are broken by too many individually reasonable hops.

Why This Architecture?

Networking is the delivery layer for every distributed system (systems where many machines cooperate to act like one). The right protocol and edge architecture reduce latency, improve availability, and keep teams from reinventing routing, authentication, and caching in every service.

Common Flow Patterns

  • Browser to web app: DNS -> CDN -> reverse proxy -> app server -> database.
  • Mobile app to backend API: DNS -> API gateway -> load balancer -> service.
  • Internal service call: service mesh (infrastructure that manages service-to-service traffic) or direct gRPC -> backend service.
  • Live updates: client opens WebSocket -> server pushes events in real time.

What Can Fail?

  • DNS misconfiguration sends traffic to the wrong endpoint.
  • TLS certificate expiration breaks HTTPS.
  • Load balancer overload or bad health checks route traffic poorly.
  • Reverse proxy misrouting creates hard-to-debug 5xx errors.
  • CDN cache miss or invalidation bugs increase origin load.
  • WebSocket connections drop during server restart or network jitter.
  • API gateway overload becomes a single bottleneck.

How Much Will It Cost?

Component Typical Cost Driver

The biggest hidden cost is usually egress bandwidth and duplicated traffic due to poor caching or chatty service-to-service calls.

How Will Teams Maintain It?

  • Use managed DNS and certificate automation where possible.
  • Track LB 5xx rates, p95 latency, and unhealthy host counts.
  • Version APIs and publish contracts so clients do not break.
  • Keep reverse proxy configs and gateway policies in source control.
  • Monitor CDN cache hit rate and origin traffic.
  • Regularly test failover, certificate renewal, and route changes.

How Does It Evolve in 3 Years?

Year Evolution
Year 1 Basic HTTP APIs behind a single load balancer and reverse proxy.
Year 2 CDN, API gateway, TLS automation, and internal gRPC adoption.
Year 3 Global routing, multi-region failover, service mesh, and edge compute for latency-sensitive paths.

Design Axes

  • Scalability: horizontal scaling usually happens at the proxy, LB, or gateway layer first.
  • Latency vs throughput (how much work the system finishes per second): gRPC and CDNs optimize latency; buffering and caching optimize throughput.
  • Availability vs consistency (whether different parts of the system see the same data at the same time): DNS and CDN caches may trade freshness for reachability.
  • Reliability and fault tolerance: redundancy at each hop prevents a single network component from taking the system down.
  • Stateless vs stateful: edge components are preferably stateless; state lives in services, caches, or durable stores.
  • Synchronous vs asynchronous: HTTP/gRPC are synchronous; WebSockets are persistent; queues are asynchronous.

Capacity Estimation

  1. DNS: Query volume is usually tiny compared with application traffic, but low TTLs and failovers can increase query rate.
  2. CDN: If 90% of static traffic is cached at the edge, origin bandwidth drops by 10x.
  3. Load balancer: A single LB can front many app servers, but TLS handshakes, open connections, and L7 routing rules still consume resources.
  4. API gateway: Auth, rate limiting, and logging add CPU and latency, so gateway capacity must be sized before app servers saturate.

Interview Trade-Off Questions

  1. When would you choose gRPC over REST for a new service boundary, and what does that choice cost you in tooling or client compatibility?
  2. If global users see high latency, would you first add a CDN, move compute closer to users, or reduce backend hops?

Chapter 3: Fundamentals of System Design

Why Use a Distributed System?

Four forces drive the move from single-node to distributed architectures:

  1. Scale — A single server has finite CPU, memory, disk, and network bandwidth. When traffic or data volume exceeds that ceiling, distributing work across many machines is the only practical option.
  2. Fault tolerance — A single server will eventually crash, lose power, or suffer a hardware failure. Distributing state and compute across multiple machines means no one failure takes down the whole system.
  3. Geographic distribution — Serving users in Tokyo from a data center in Virginia adds ~150ms of round-trip latency. Distributing compute and data to regions near users cuts that to single-digit milliseconds.
  4. Independent scalability — Different parts of a system have different bottlenecks. The search index needs massive storage and read throughput. The ads system needs fast write throughput. A distributed architecture lets each component scale on its own axis.

The Fundamental Tension

Distributed systems are hard precisely because of what happens over a network that does not happen on one machine:

  • Messages can be delayed or lost — a node cannot distinguish between a slow reply and a dead node.
  • Clocks drift — two servers' clocks are rarely in perfect sync, making "happened before" a non-trivial question.
  • Partial failures — one part of the system can fail while the rest keeps running, leaving the system in an ambiguous state.
  • Concurrent updates — two nodes can modify the same piece of data at the same time, creating conflicts.

These properties are captured formally in two foundational ideas: the CAP theorem (a distributed system can guarantee at most two of Consistency, Availability, and Partition tolerance at once — see Chapter 5) and the fallacies of distributed computing (the eight false assumptions engineers commonly make, such as "the network is reliable" and "latency is zero").

Property What it means Example

How Nodes Coordinate

Nodes in a distributed system coordinate through one of three broad patterns:

  • Leader-follower (primary-replica) — One node is elected as the authoritative source for writes. All other nodes replicate its state. Simple to reason about but the leader is a single point of failure if not handled carefully. Used by: Postgres, Redis (in sentinel mode), Kafka topic partitions.
  • Leaderless (peer-to-peer) — Any node can accept a write, and writes propagate to other nodes. Requires conflict resolution (last-write-wins, vector clocks, or CRDTs). Resilient to node loss but harder to make strongly consistent. Used by: Cassandra, DynamoDB, Amazon S3.
  • Consensus-based — A cluster of nodes votes on every decision. A write is committed only after a majority agrees. Provides the strongest consistency guarantees but is slower and limited to small cluster sizes (3 or 5 nodes). Used by: etcd, ZooKeeper, Raft-based systems like CockroachDB.

Failure Modes Unique to Distributed Systems

Failure Mode Description Example
Network partition Two groups of nodes cannot communicate with each other An AWS availability zone loses connectivity; nodes inside still work but cannot reach nodes in other AZs
Split-brain Two nodes both believe they are the leader and accept conflicting writes A fencing token or lease expiry is needed to ensure only one writer proceeds
Byzantine fault A node sends incorrect or malicious information to others Common in blockchain networks; handled via Byzantine fault-tolerant (BFT) consensus
Clock skew Nodes disagree on the current time Spanner uses TrueTime (GPS + atomic clocks) to bound clock uncertainty; most systems use logical clocks instead
Cascading failure One slow node causes its callers to queue up, overloading the whole system Circuit breakers and timeouts at every service boundary prevent one slow dependency from dragging down everything

Real-world example: In 2012, AWS experienced a widespread outage when an EBS control plane issue caused a cascading failure across services. Correctly isolated distributed systems (those with circuit breakers and async decoupling) degraded gracefully; tightly coupled ones went down entirely.

When Not to Use a Distributed System

Distribution solves scale and fault-tolerance problems, but it introduces complexity, latency, and operational overhead. A distributed system is not always the right answer:

  • If your traffic fits on one well-sized machine (most startups at < 10K RPS), a single-node deployment with a managed backup is far simpler to operate.
  • If your data fits in one database with a read replica, you do not need sharding, leaderless replication, or consensus clusters.
  • If strong consistency is required everywhere and you cannot tolerate the performance cost of cross-node coordination, a well-tuned monolith may be the correct choice.

The rule of thumb: distribute only what must be distributed, and only when the cost of not distributing is real and measurable.

The Core Forces Behind Most System Designs

Nearly every design decision in distributed systems is a response to a small number of recurring forces:

  • Latency (how long one request or operation takes) — how long one operation takes from the user's point of view.
  • Throughput (how much work the system finishes per second) — how much work the system can complete per second.
  • Availability (how often the system stays usable) — whether the system keeps serving requests during failures.
  • Consistency (whether different parts of the system see the same data at the same time) — whether different parts of the system agree on the same value at the same time.
  • Durability (whether acknowledged data survives crashes) — whether acknowledged data survives crashes and recovery.
  • Operability (how easy the system is to run, debug, and change safely) — whether humans can debug, deploy, and evolve the system safely.

Those forces interact. Batching improves throughput but can hurt latency. Stronger consistency improves correctness but may reduce availability during partitions. More caching reduces latency and cost, but increases staleness risk. Sharding (splitting data across many machines) improves scale, but complicates joins, transactions (groups of operations that succeed or fail together), and debugging.

That is why "best design" is rarely absolute. Search prefers low latency and can tolerate slightly stale results. A bank ledger prefers durability and correctness over raw write throughput. A metrics pipeline often prefers availability and eventual convergence over strict transactional guarantees. When a system-design answer feels vague, it is usually because the speaker has not yet said which force matters most.

Microservices

The microservice architecture structures an application as a collection of small, independently deployable services. Each service:

  • Is organized around a specific business capability
  • Is owned by a small team (the "two-pizza rule": a team small enough to be fed by two pizzas)
  • Is independently deployable and scalable without coordinating with other teams
  • Is loosely coupled from other services — changes inside one service do not require changes in others
  • Is highly maintainable and testable in isolation
  • Owns its own data — no shared databases between services

Real-world example: Netflix decomposes its platform into hundreds of microservices — user auth, recommendations, billing, video encoding, content delivery, and more. A bug in the recommendations service does not bring down billing. Each service is deployed dozens of times per day by its own team.

Monolith vs. Microservices: When to Choose Each

Before reaching for microservices, it is worth understanding what a monolith is and when it is the better option.

A monolith is a single deployable unit where all business logic, data access, and API handling live together. It is faster to develop initially, trivial to test end-to-end, and has zero network overhead between components. The right choice for teams under ~15 engineers, products in early discovery, and systems where operational simplicity matters more than independent scalability.

A modular monolith is an intermediate option: code is organized into clearly separated modules with defined interfaces, but it is still deployed as one unit. This is often the right answer for teams that have not yet found their true service boundaries but want clean code organization.

Microservices become the right choice when:

  • Different parts of the system have genuinely different scaling requirements (the video encoding pipeline needs 100× more CPU than the user auth service).
  • Teams are large enough that a shared codebase creates coordination overhead and deploy bottlenecks.
  • Parts of the system need different technology stacks (ML inference in Python, low-latency trading logic in C++, web APIs in Go).
  • Blast radius from failures must be contained — a bug in one service should not take down unrelated capabilities.

How to Split Services: The Five Axes

Poor service boundaries are the most common microservices mistake. Use these axes to decide where to draw lines:

  1. Business capability — split around what the business does, not how the code is organized technically. "Order Management," "Payment Processing," "User Profile" are good boundaries. "Database Layer" or "Utility Functions" are bad ones.
  2. Data ownership — each service should own its primary data store exclusively. If two services need to join data at the database level, they are probably too fine-grained.
  3. Change cadence — a component that changes frequently (product catalog) should not be coupled to one that rarely changes (billing rules). Different change rates justify different services.
  4. Scaling profile — components with different resource needs (CPU, memory, I/O) benefit from independent scaling. Video transcoding and user authentication have nothing in common operationally.
  5. Transaction boundary — if two operations must always succeed or fail together atomically, keeping them in the same service is simpler than coordinating a distributed transaction. Split across services only when the business can tolerate eventual consistency between them.

Inter-Service Communication

Services communicate in one of two ways, and choosing the right one for each interaction is critical:

Synchronous (request-response):

  • The caller waits for a response before proceeding.
  • Used for: queries where the result is needed immediately (fetch user profile, check inventory).
  • Protocols: REST over HTTP, gRPC (preferred for internal calls — binary encoding, streaming, strong contracts).
  • Risk: the caller is blocked while the callee processes. If the callee is slow, the caller slows down too — creating latency amplification across chains of services.

Asynchronous (event-driven):

  • The caller publishes an event and moves on. Downstream services consume it at their own pace.
  • Used for: operations where the caller does not need an immediate result (send email after order placed, update search index after product updated).
  • Protocols: Kafka, Amazon SQS, RabbitMQ.
  • Benefit: temporal decoupling — the producer and consumer do not need to be running simultaneously. The queue absorbs bursts.
  • Risk: eventual consistency — the downstream service may lag behind the producer by seconds or minutes.

The rule of thumb: use synchronous calls for queries (read operations) and async events for commands that trigger side effects in other services (write operations with fan-out).

Key Resilience Patterns

When services call each other synchronously, failures cascade unless you actively contain them. These four patterns are the standard toolkit:

Circuit Breaker — a wrapper around outbound calls that tracks the error rate. If errors exceed a threshold (e.g., 50% of calls in the last 10 seconds fail), the circuit "opens" and subsequent calls fail immediately without waiting for a timeout. This prevents a slow downstream service from holding threads and connections in the calling service. After a cooldown period, the circuit allows a probe request through to check if the downstream has recovered. Implemented by: Resilience4j (Java), Polly (.NET), or at the infrastructure layer via a service mesh (Istio, Linkerd).

Timeout — every outbound call must have a deadline. Without a timeout, a hung downstream service will hold a thread in the caller indefinitely, eventually exhausting the thread pool. A common mistake is setting timeouts only at the outermost API gateway while inner service-to-service calls have none.

Retry with exponential backoff and jitter — transient failures (network blip, brief overload) are common in distributed systems. Retrying the same call after a short delay often succeeds. Exponential backoff prevents retry storms: wait 100ms, then 200ms, then 400ms. Jitter (adding randomness to the wait time) prevents all retrying clients from hitting the recovering service at exactly the same moment.

Bulkhead — isolate resources so that one slow integration does not exhaust resources shared with other integrations. Example: use a separate thread pool for calls to the payment service, so a payment outage does not exhaust threads needed for the inventory service. Named after the watertight compartments in a ship's hull.

The Saga Pattern: Distributed Transactions Without 2PC

Microservices with separate databases cannot use a traditional database transaction that spans services. The naive alternative — distributed two-phase commit (2PC) — is slow, fragile, and creates tight coupling between services. The better solution is the Saga pattern.

A Saga is a sequence of local transactions, each in its own service, where each step publishes an event that triggers the next step. If a step fails, the Saga executes compensating transactions to undo the completed steps.

Example: placing an e-commerce order

  1. Order Service creates the order (status: PENDING) → publishes OrderCreated event.
  2. Inventory Service reserves the items → publishes InventoryReserved event.
  3. Payment Service charges the card → publishes PaymentProcessed event.
  4. Order Service marks the order CONFIRMED.

If payment fails at step 3:

  • Payment Service publishes PaymentFailed event.
  • Inventory Service receives it and releases the reserved items (compensating transaction).
  • Order Service marks the order CANCELLED.

Two Saga execution styles:

  • Choreography — each service listens for events and reacts. No central coordinator. Simple to implement but hard to trace when a flow goes wrong across many services.
  • Orchestration — a dedicated Saga Orchestrator service explicitly calls each step and handles compensations. Easier to visualize and debug but introduces a central component that must be kept reliable.

Data Management in Microservices

The database-per-service pattern is non-negotiable for true independence, but it creates challenges that a shared database does not have.

Database-per-service: Each service has its own schema, and no other service queries it directly. Cross-service data needs are satisfied through API calls or events. This means:

  • The Order Service cannot JOIN against the User Service's table — it must call the User Service's API or maintain a local copy of the data it needs.
  • Schema changes in one service do not break others.
  • Each service can choose the right storage technology for its workload (Postgres for transactional data, Cassandra for write-heavy time-series, Redis for ephemeral session state).

The CQRS pattern (Command Query Responsibility Segregation): Separate the data model used for writes (commands) from the data model used for reads (queries). The write model optimizes for correctness and consistency; the read model optimizes for query performance (denormalized, pre-joined, cached). Changes on the write side are propagated to the read side asynchronously via events. This is especially useful when a service's read patterns are very different from its write patterns (e.g., a social feed where writes are per-user but reads are per-follower).

The Outbox Pattern: When a service needs to update its own database and publish an event atomically, it writes both the data change and the event to the same local database in a single transaction. A separate process (the "outbox publisher") reads unpublished events from the outbox table and forwards them to the message broker. This guarantees that events are never lost even if the broker is temporarily unavailable at write time.

Service Discovery

In a dynamic environment where services start, stop, and scale horizontally, hardcoding IP addresses is not viable. Service discovery solves this.

  • Client-side discovery: The client queries a service registry (Consul, etcd, Eureka) to get the list of healthy instances and picks one using a load balancing algorithm. The client owns the routing logic.
  • Server-side discovery: The client sends requests to a load balancer or API gateway, which queries the registry and forwards to a healthy instance. The client has no knowledge of the registry.
  • Kubernetes DNS: In Kubernetes, each service gets a stable DNS name (e.g., payment-service.payments.svc.cluster.local). The kube-proxy layer handles routing to healthy pods transparently. This is the most common form of service discovery for containerized microservices.

API Gateway

An API gateway is the single entry point for all external traffic into a microservices system. It handles concerns that should not live in every individual service:

  • Authentication and authorization — validate JWT tokens, enforce OAuth scopes.
  • Rate limiting — protect backend services from overload by limiting calls per client.
  • Request routing — forward /api/orders/* to the Order Service and /api/users/* to the User Service.
  • Protocol translation — accept REST from external clients, translate to gRPC for internal services.
  • Response aggregation — combine responses from multiple services into a single API response (the Backend-for-Frontend pattern).
  • Observability — log every request, emit metrics, inject trace IDs.

Common tools: AWS API Gateway, Kong, Nginx, Envoy. The API gateway should be thin — it routes and enforces policies, but business logic stays in the services.

Why this architecture?

A monolith is fast to start with, but as the codebase grows past ~50 engineers, a single deploy cycle becomes a coordination nightmare — one broken test blocks the whole release train. Microservices give each team an independent deploy pipeline. At Amazon, the move to services was driven by a 2002 Jeff Bezos mandate: all data and functionality must be exposed via service interfaces, with no other form of inter-process communication.

The deeper reason microservices work at scale is Conway's Law: organizations tend to produce architectures that mirror their communication structures. A company with 20 product teams building a monolith spends enormous energy coordinating changes. The same company with well-defined service boundaries lets each team move at its own pace.

Load Balancers

A load balancer sits between clients and a pool of backend servers. Every incoming request arrives at the LB, which picks a healthy backend, forwards the request, and streams the response back to the client. From the client's perspective it looks like a single server; from the backend's perspective, load is spread evenly across the whole fleet.

Client ──► Load Balancer ──► Server A
                         ├──► Server B
                         └──► Server C

Without a load balancer, all traffic hits one server. A single modern server handles roughly 10 K–100 K requests/second depending on workload complexity. A load balancer lets you add more servers and distribute load automatically — this is horizontal scaling. It also provides built-in failover: the moment a backend fails its health check, the LB drains its connections and stops routing new traffic to it, usually within 10–30 seconds.

Layer 4 vs. Layer 7

The OSI model defines networking layers. Load balancers operate at either Layer 4 (Transport) or Layer 7 (Application), and the choice has real performance and routing implications.

Layer 4 (Transport) Layer 7 (Application)
Routes on IP address + TCP/UDP port HTTP headers, URL path, query params, cookies
TLS termination No (passes encrypted bytes through) Yes (decrypts, inspects, re-encrypts)
Content-based routing No Yes (/api/video/* → video fleet, /api/auth/* → auth fleet)
Throughput Very high — no packet inspection overhead Slightly lower — parses HTTP on every request
Typical use Database proxies, raw TCP services, gaming Web apps, REST/gRPC APIs, microservices
AWS equivalent NLB (Network Load Balancer) ALB (Application Load Balancer)

When to pick L4: You need maximum throughput and lowest latency, you are proxying a non-HTTP protocol (e.g., MySQL, Redis, MQTT), or you cannot afford the overhead of TLS termination at the LB.

When to pick L7: You need path-based routing, host-based routing, header rewriting, sticky sessions, WAF (Web Application Firewall) integration, or detailed HTTP access logs.

Load Balancing Algorithms

The algorithm decides which backend receives the next request. Choosing the wrong one causes hot spots — one backend saturated while others idle.

Algorithm How it works Best for Watch out for
Round Robin Cycles through backends in order (A → B → C → A → …) Homogeneous fleets where every request costs roughly the same Uneven load if some requests are much heavier
Weighted Round Robin Like round robin but higher-capacity servers receive proportionally more requests Mixed instance types (e.g., c5.2xlarge gets 2× weight of c5.large) Weights must be re-tuned when fleet changes
Least Connections Routes to the backend with the fewest active open connections at that moment Long-lived connections such as WebSockets, database proxies, file uploads Slightly higher LB CPU to maintain connection counters
Least Response Time Routes to the backend that replied fastest recently Latency-sensitive APIs where backend speed varies by request type Slow backends starved entirely if one backend is always fastest
IP Hash Hashes the client IP to always map to the same backend Stateful workloads where session state is stored locally Backend failure changes hash mapping, breaking all sessions
Random with Two Choices (Power of Two) Picks two backends at random, routes to the least loaded of the pair Very large fleets where centralized state tracking is expensive Slightly worse distribution than Least Connections at small scale

Rule of thumb: Default to Least Connections for general web workloads. Switch to Weighted Round Robin when your fleet is heterogeneous. Avoid IP Hash for stateful session storage — externalize session state to Redis instead.

TLS Termination

HTTPS traffic is encrypted end-to-end using TLS (Transport Layer Security). The load balancer can handle TLS in three ways:

  1. TLS Termination (most common): The LB decrypts traffic, routes in cleartext to backends on the private network. Backends do not need TLS certificates. Pros: cheaper backends, full L7 visibility. Cons: traffic inside the datacenter is unencrypted (acceptable on a trusted private network).

  2. TLS Passthrough: The LB forwards encrypted bytes without decrypting. Backends terminate TLS themselves. Pros: end-to-end encryption. Cons: L7 routing is impossible since the LB cannot read the HTTP headers.

  3. TLS Re-encryption: The LB terminates, inspects, then re-encrypts before forwarding to backends. Full end-to-end encryption and L7 routing. Cons: double TLS overhead.

Certificate management is the most common operational pain point. Automate renewal with AWS Certificate Manager (ACM) or Let's Encrypt with cert-manager — never manage certificates manually.

Health Checks

The LB continuously probes backends to detect failures before they affect users.

Health check type How it works When to use
TCP Opens a connection; success if the port accepts Simple, low overhead; does not verify app is healthy
HTTP/HTTPS Sends a GET to a path (e.g., /health); success if response is 2xx Recommended for web services — verifies app logic is up
Custom script Runs a script on the backend Rare; used for complex startup readiness checks

Health check parameters to configure:

  • Interval: How often to probe (e.g., every 10 s).
  • Timeout: How long to wait for a response (e.g., 5 s).
  • Healthy threshold: Consecutive successes before marking healthy (e.g., 2).
  • Unhealthy threshold: Consecutive failures before marking unhealthy (e.g., 3).

A backend marked unhealthy has its connections drained (in-flight requests complete; new requests stop) and then removed from rotation. This is called connection draining or deregistration delay (AWS default: 300 s).

Design your /health endpoint to check real dependencies — a 200 response that ignores a broken database connection is useless. Return 200 only when the service can actually serve traffic.

Sticky Sessions (Session Affinity)

By default, each request can land on any backend. Sticky sessions pin a client's requests to the same backend using a cookie (L7) or source IP (L4).

Why you might use it: Legacy apps that store session state in memory on the server (shopping cart, login token) cannot handle requests hopping between backends.

Why you should avoid it:

  • If the pinned backend crashes, the user's session is lost.
  • Traffic distribution becomes uneven as some backends accumulate sticky connections.
  • It prevents true horizontal scaling.

Better approach: Externalize session state to a shared store (Redis, DynamoDB). Now every backend is stateless and can serve any user's request — no stickiness needed.

Global Load Balancing

A single regional LB routes traffic within one datacenter region. Global load balancing adds a layer above that to route users to the nearest healthy region.

User in Tokyo ──► Global LB ──► Asia-Pacific Region
User in NYC   ──► Global LB ──► US-East Region
User in London ──► Global LB ──► EU-West Region

Options:

  • AWS Global Accelerator: Anycast IPs that route to the nearest AWS region over the AWS backbone, cutting latency significantly vs. the public internet.
  • Cloudflare Load Balancing: Routes to origin based on latency, geography, and health checks. Also benefits from Cloudflare's CDN and DDoS protection.
  • Route 53 Latency-Based Routing: DNS-level routing to the lowest-latency region. Simple but subject to DNS TTL delay on failover.

Use global load balancing when you have users across multiple continents or need sub-100 ms latency globally.

Real-world example: During Amazon Prime Day, the AWS ALB fleet absorbs hundreds of millions of requests per second globally. Each regional ALB autoscales its own capacity, while Route 53 and Global Accelerator handle cross-region routing. A backend fleet of thousands of EC2 instances uses Least Connections routing so that heavier requests (large product pages) do not pile onto the same servers.

Why this architecture?

Without a load balancer, all traffic hits a single server. One server caps out at ~10–100K requests/second depending on complexity. A load balancer enables horizontal scaling — add more servers, the LB distributes load automatically. It also provides health checking — if server 3 fails, the LB stops sending traffic to it within seconds.

Databases

A database provides structured, durable storage with access control and consistency guarantees.

ACID (transaction guarantees for correctness and crash safety) guarantees:

  • Atomicity — A transaction succeeds entirely or not at all. (Bank transfer: deduct A and credit B must both succeed.)
  • Consistency — Data transitions from one valid state to another. (A balance cannot go negative if the constraint disallows it.)
  • Isolation — Concurrent transactions do not interfere. (Two users buying the last item simultaneously — only one succeeds.)
  • Durability — Committed data survives crashes. (Data flushed to disk before acknowledging the commit.)

SQL (the relational database language and ecosystem used by systems like Postgres and MySQL) vs NoSQL (non-relational data stores built for flexibility or scale):

Feature SQL (Postgres, MySQL) NoSQL (Cassandra, MongoDB, DynamoDB)
Schema Fixed, structured Flexible, schema-less
Scaling Vertical (primarily) Horizontal
Transactions Full ACID Eventual consistency (a model where replicas may differ briefly but converge later) (usually)
Query model Rich SQL joins, aggregations Key-value, document, or column scans
Best for Financial records, user profiles, joins High write throughput, flexible/dynamic data, global scale

Real-world example: Stripe uses Postgres for its core payments ledger (ACID required) but uses Redis for rate-limit counters and DynamoDB for audit log event streams (write-heavy, no joins needed).

Why this architecture?

SQL databases are the right default when data has relationships, requires consistent multi-row updates, and needs complex queries. NoSQL is the right choice when write throughput exceeds what a single SQL primary can handle (typically >100K writes/second), when the data model is inherently key-based, or when global distribution with low write latency is required (Cassandra's leaderless design handles multi-region writes natively).

Example decision point: Instagram's user feed metadata is stored in Cassandra (high write throughput, simple key lookups), but the user's account settings are in Postgres (relational, low write volume, requires consistency).

Caches

A cache is a small, fast storage layer that reduces latency and database load by serving frequently accessed data from memory.

Cache levels:

Level Example What it caches
Client cache Browser cache HTML, CSS, JS, images
CDN (content delivery network, edge servers that cache content near users) cache Cloudflare, CloudFront Static assets served from edge nodes near the user
Web server cache NGINX proxy cache Full HTTP responses
Database cache Query result cache (legacy / rare; prefer application-level caching) Query result sets
Application cache Redis, Memcached DB query results, computed objects, session tokens

Real-world example: Twitter uses Redis to cache pre-computed timelines for active users. Without the cache, loading a timeline would require a multi-join query across hundreds of millions of tweets and follow relationships — taking seconds per load.

Why this architecture?

Applications follow a Zipfian distribution (a skewed pattern where a small set of items gets most of the traffic, often roughly 80/20) — 80% of requests touch the same 20% of data. Caches exploit this by keeping hot data in fast memory (~100 ns access vs. ~10ms for disk). The result: a cache hit rate of 99% means only 1 in 100 requests hits the database, enabling a single DB instance to serve 100× more users.

Example: Facebook's Memcached deployment absorbs billions of cache reads per second, preventing those requests from ever reaching MySQL.

Messaging Queues

Messaging queues enable asynchronous communication between services. Producers write messages to the queue; consumers read and process them independently and at their own pace.

Core components:

  • Producer — The service that creates and publishes messages (e.g., the Uber Ride Service publishes a RIDE_CANCELLED event).
  • Consumer — The service that reads and processes messages (e.g., the Notification Service consumes RIDE_CANCELLED to send a push notification).
  • Broker (the system that stores and routes messages) — The messaging infrastructure that stores and routes messages (Kafka, RabbitMQ, SQS).

Popular implementations: Apache Kafka, RabbitMQ, Amazon SQS, Google Pub/Sub (publish-subscribe, where one event can go to many consumers).

Real-world example: Uber uses Kafka to propagate all ride lifecycle events (booking, driver location updates, cancellation) between services. The booking service does not wait for the notification service — it fires the event and moves on.

Why this architecture?

Synchronous HTTP calls between services create tight coupling and cascading failure risk. If the Notification Service is slow or down, a synchronous call from the Booking Service would cause the booking itself to timeout. A queue decouples them: the Booking Service writes to Kafka and returns immediately; the Notification Service processes the event whenever it is ready. This provides:

  1. Temporal decoupling — Producer and consumer do not need to be running simultaneously.
  2. Load leveling — A burst of 1M bookings in 10 seconds can be drained from the queue over the next 30 seconds by the consumer, rather than overwhelming it instantly.
  3. Replay — Kafka retains messages for configurable periods (days/weeks). If the Notification Service had a bug and dropped messages, you can replay the Kafka topic from a past offset.

How Good Service Boundaries Are Chosen

The hardest part of microservices is not deployment technology. It is choosing boundaries that remain stable as the product changes.

Good service boundaries usually align around five things:

  1. Business capability — one service owns one coherent domain such as identity, catalog, checkout, or search.
  2. Data ownership — the service is the source of truth for its state. Other services read through APIs or events, not by reaching into the database.
  3. Change cadence — the team can evolve its code and schema (the structure and rules of stored data) without constantly coordinating with unrelated teams.
  4. Scaling profile — hot paths that need very different compute or storage characteristics deserve separate scaling boundaries.
  5. Transaction (groups of operations that succeed or fail together) boundary — if two pieces of logic must commit atomically all the time, splitting them into separate services too early usually creates more pain than value.

Bad boundaries show up quickly. If every page render fans out through six services, the split is too chatty. If every feature requires schema changes across four databases, the boundaries are not independent. If one huge "user service" owns identity, profiles, privacy, social graph, and billing, the boundary is too broad to scale cleanly.

This is why many teams should begin with a modular monolith. A well-structured monolith can preserve domain boundaries inside one codebase while postponing the operational cost of microservices. Split a service when the boundary is already visible in ownership, data, scaling, or reliability needs, not because "microservices" sounds modern.

Why this architecture?

The key driver is independent deployability. In a monolith, all 200 engineers merge code into one codebase; any broken test or merge conflict blocks the entire release train. In microservices, the Reviews team can deploy 10 times per day without waiting for the Payments team.

The second driver is independent scalability. During Prime Day, Amazon needs to scale its Order Processing service 100× but its Returns service only 2×. Microservices allow this; a monolith scales as a single unit.

What can fail?

The same table from Chapter 3 applies, plus:

  • Service proliferation — Too many services (~100+) without a service catalog creates "where is this logic?" confusion.
  • Distributed transaction failures — A multi-service transaction (e.g., place order → deduct inventory → charge card) that partially fails leaves data inconsistent. Solution: Saga pattern (a multi-step workflow that uses local transactions plus compensating actions) (choreography or orchestration) with compensating transactions.
  • Latency (how long one request or operation takes) amplification — A user request that chains through 5 services in series adds the latency of each hop. Solution: parallelize independent calls; set strict timeouts.

How much will it cost?

Microservices cost more than a monolith at small scale:

  • More infrastructure (one DB per service vs. one shared DB).
  • More CI/CD pipelines to maintain.
  • More observability tooling to correlate cross-service traces.

Rough multiplier: A comparable monolith might cost $500/month; the microservices equivalent costs $2,000–$5,000/month initially. The crossover point — where the operational efficiency gains offset cost — typically occurs at 20–30 engineers or 5M+ users.

How will teams maintain it?

  • API versioning (e.g., /v1/orders) prevents breaking changes when a service evolves its contract.
  • Contract testing (Pact, Spring Cloud Contract) — consumers define what they expect from a provider's API; provider tests verify it before deploy.
  • Service ownership matrix — every service has a named owner, an SLO (service level objective, the target value for a reliability metric), and an on-call rotation.
  • Incident correlation — when a user reports "checkout is broken," teams need distributed traces to identify which of the 5 services in the chain failed.

Principle 1: Domain-Driven Design

Applications are organized around business domains. Each domain has a bounded context — its own vocabulary, data model, and team ownership.

Example — Credit Card Processing System:

  • Merchant Subdomain — Merchant acceptance workflow, merchant data.
  • Accounting Subdomain — End-of-day clearing and reconciliation.
  • Payment Subdomain — Online transaction processing.

Bounded Context example: The word "Customer" means different things:

  • In the Sales context: a prospect I am trying to convert.
  • In the Support context: a user who has opened a ticket.

Each context models Customer differently. The Sales service's Customer object should not be shared directly with the Support service. They exchange only what is needed via a defined API contract.

Real-world example: Stripe's public API is domain-driven — /charges, /subscriptions, /customers, /invoices each correspond to an internal bounded context with its own team, data store, and deployment pipeline.

Why this architecture?

Without DDD, service boundaries are drawn arbitrarily (e.g., by technical layer: "frontend service", "backend service") rather than by business capability. This leads to services that change together, defeating the purpose of independent deployability. DDD ensures service boundaries align with organizational ownership — Conway's Law states that a system's architecture mirrors the communication structure of the organization that built it.

What can fail?

  • Incorrect bounded context — If the Order and Inventory contexts are merged too early, a deploy of Inventory logic can break the Order service. Solution: revisit domain boundaries after 6 months of production use.
  • Leaky abstraction — Service A exposes internal DB fields in its API. When A refactors its DB, B's API contract breaks. Solution: API versioning + consumer-driven contract tests.

How will teams maintain it?

  • Event storming workshops (Domain-Driven Design discovery technique): bring product, engineering, and business together to map events, commands, and aggregates on a whiteboard. This surfaces domain boundaries early.
  • Assign domain experts to each bounded context team — the payments team should include someone who understands payment processing, not just software engineers.

How does it evolve in 3 years?

Bounded contexts become more refined over time. What starts as a single Orders service often splits into OrderCreation, OrderFulfillment, and OrderReturns as the team discovers distinct change rates and ownership boundaries. DDD is an ongoing process, not a one-time design exercise.

Principle 2: Culture of Automation

Microservice ecosystems require automation at every layer to be operationally sustainable.

  • Infrastructure as Code (IaC) — Tools like Terraform, Pulumi, and AWS CDK define infrastructure in code. A new environment (staging, DR) is spun up from code in minutes, not weeks of tickets.
  • Testing Automation — JMeter, Gatling (load tests), Cypress (E2E), contract tests (Pact). Services must be tested in isolation before release.
  • Continuous Delivery (CD) — Harness, Spinnaker, GitLab CI/CD. Each service can be deployed independently without human approval (for non-critical paths).

Real-world example: Netflix deploys hundreds of times per day using Spinnaker. A single engineer can safely deploy a microservice that serves 250M users. Without automation, this would require a coordinated release process involving dozens of teams.

Why this architecture?

Manual processes do not scale with the number of services. If deploying 100 services takes 1 hour each, you need 100 hours of human time per release cycle. Automation compresses this to minutes. IaC also eliminates "works in staging, broken in production" drift — the infrastructure definition is version-controlled and identical across environments.

Principle 3: Encapsulation

Each service exposes behavior through a well-defined API and hides its internal implementation (DB schema, business logic, auth mechanism).

Bad design: Multiple services share a single database. When the Orders team renames a column in the orders table, the Inventory and Payments teams' code breaks silently at runtime.

Good design: Each service owns its database. Schema changes are internal. External access is only through versioned API contracts.

Bounded Contexts and APIs: APIs must reflect the context they serve, not leak internal state. The Support API's Customer response should include ticket_count but not sales_pipeline_stage — that belongs to the Sales context.

Real-world example: Shopify's merchant API exposes order data through /orders without revealing whether the backend uses MySQL, Kafka, or a data warehouse internally. Shopify has changed its storage layer multiple times without breaking any of the thousands of apps built on its API.

Why this architecture?

Encapsulation is the only mechanism that enables independent deployability. If Service A's internal DB schema leaks to Service B, a schema change in A requires a coordinated deploy of both A and B — the exact coupling that microservices are meant to eliminate.

What can fail?

  • API backward-incompatible change — Removing a field from a response breaks all clients consuming that field. Solution: additive-only API changes; deprecation notices 60 days before removal.
  • Data leakage — Sensitive fields (PII, financial data) accidentally included in a public API response. Solution: API response schemas explicitly allowlist fields; never pass internal DB objects directly to serializers.

How will teams maintain it?

  • Maintain an API changelog and versioning policy (e.g., /v1, /v2).
  • Use OpenAPI (Swagger) specs to document and validate request/response schemas.
  • Consumer-driven contract tests (Pact) — consumers publish what they expect; provider CI runs these tests before any deploy.

Schema Evolution and Backward Compatibility

Schema changes are the most common source of silent production breakage in microservice systems. Three rules govern safe evolution:

  1. Additive-only changes are safe. Adding a new optional field to a JSON response or Protobuf message does not break existing consumers that ignore unknown fields.
  2. Removing or renaming fields is breaking. Any consumer relying on user.email_address breaks silently if it becomes user.email. Use field deprecation with a 30–90 day grace window before removal.
  3. Type changes are breaking. Changing a field from int to string (or adding/removing null-ability) breaks deserialization for all consumers unless done behind a new API version.

Safe migration pattern (expand-contract):

  1. Expand: Add the new field alongside the old one. Both exist in the response.
  2. Migrate: Update all consumers to use the new field.
  3. Contract: Remove the old field once all consumers have deployed.

For event schemas on Kafka or other message buses, use a Schema Registry (Confluent, AWS Glue) that enforces compatibility rules (BACKWARD, FORWARD, or FULL) before a producer is allowed to publish a new schema version. This prevents a bad schema from reaching consumers before they are updated.

Principle 4: Decentralization

Teams need autonomy to build and operate their services at speed. Decentralization occurs at two levels:

  • Self-Service Infrastructure — Teams provision resources independently via cloud consoles and IaC, without waiting for central IT approvals (which can take days or weeks).
  • Shared Governance — Instead of one central architecture team mandating a single tech stack, each team picks tools that fit the job (polyglot persistence, multiple languages), while aligning on cross-team standards (API contracts, security policies, observability).

Real-world example: Spotify's squad model — each squad (small team) is autonomous over its domain. Guilds (horizontal communities of practice) share standards without enforcing them top-down.

Why this architecture?

Centralized governance creates bottlenecks. If every infrastructure change requires an approval from a central team, the central team becomes a constraint on the entire engineering organization. Decentralization scales linearly with the number of teams.

What can fail?

  • Technology fragmentation — Without any governance, teams use 15 different messaging systems, making cross-team data sharing impossible. Solution: defined technology radar (Thoughtworks-style) listing approved, trial, and deprecated tech.
  • Security inconsistency — Teams implement auth differently. Solution: shared auth middleware or sidecar proxy (mTLS via Istio), not each team rolling their own.

Principle 5: Smart Endpoints, Dumb Pipes

Business logic lives inside microservices (smart endpoints), not in the communication layer (dumb pipes).

SOA's ESB model (anti-pattern): The Enterprise Service Bus contains routing logic, data transformations, orchestration, and business rules. Changing any business rule requires modifying the ESB — which is owned by a central team and changes slowly.

Microservices model: Kafka or HTTP is a dumb pipe — it delivers bytes without understanding them. The consuming service interprets and acts on the message. Business logic changes require only a deploy of the consuming service, not the messaging infrastructure.

Real-world example: Netflix uses Kafka as a dumb pipe. The event VIDEO_PLAYBACK_STARTED is emitted to a Kafka topic. Downstream services (analytics, recommendations, billing) each consume it and apply their own logic independently. None of this logic lives in Kafka.

What can fail?

  • Logic duplication — If multiple services implement the same transformation independently, they can drift. Solution: shared libraries for truly cross-cutting logic (e.g., a shared auth validation library). But use sparingly — shared libraries create coupling.

These principles are meant to be applied together: DDD shapes the boundaries, automation makes the system operable, encapsulation preserves contracts, decentralization keeps teams unblocked, and smart endpoints keep business logic out of the pipe.

Chapter 5: Availability & CAP Theorem

The CAP Theorem

The CAP Theorem (Brewer's Theorem, 2000) states that a distributed data system can guarantee only two of the following three properties simultaneously:

  • Consistency (C) — All nodes (individual machines or service instances in a larger system) return the same, most recent data at the same time. No stale reads.
  • Availability (A) — Every request to a non-failing node gets a response within a reasonable time, even if some nodes are down.
  • Partition Tolerance (P) — The system continues operating despite network partitions (message loss or failure between nodes).

Since network partitions are unavoidable in real distributed systems (systems where many machines cooperate to act like one) (networks fail, cables are cut, packets are dropped), the real trade-off is between C and A.

Type Guarantees Examples When to choose
CA Consistency + Availability Single-node RDBMS (Postgres, MySQL) Non-distributed systems; no network partitions possible
CP Consistency + Partition Tolerance MongoDB, HBase, Redis, ZooKeeper Financial data; inventory counts; any domain where stale reads are incorrect
AP Availability + Partition Tolerance Cassandra, CouchDB, DynamoDB User activity feeds, product recommendations, any domain where eventual consistency (a model where replicas may differ briefly but converge later) is acceptable

What CAP Means In Practice

CAP is often taught too abstractly. In production, the key idea is simpler: once a partition exists, a distributed system cannot guarantee both immediate agreement and uninterrupted service for the same operation.

Two clarifications matter:

  1. Availability is per operation, not a global personality trait. A system might be CP for account-balance writes, AP for recommendation updates, and "slightly stale but very available" for search.
  2. Consistency in CAP is stronger than "the system is usually correct." It refers to linearizable behavior: every client observes the same latest write order as if there were one single machine.

That means CAP is not asking whether you "like consistency." It is asking what the system should do during a partition:

  • CP choice: reject or delay the operation until quorum (the minimum number of replicas that must agree) is restored. Safer, but less available.
  • AP choice: accept work on both sides and reconcile later. More available, but now you need merge rules, conflict handling, or compensating actions.

Most large systems are mixed. A retail site might be CP for payment authorization, AP for likes and activity feeds, and "stale for a few seconds is fine" for search. That is also why PACELC is useful in spirit: even when there is no partition, systems still choose between lower latency (how long one request or operation takes) and stronger coordination.

Proof intuition: S1 and S2 are two nodes. A network partition (a failure where machines cannot communicate with each other) occurs — they cannot communicate. A client writes to S1, then reads from S2. Either:

  • S2 returns stale data → consistency sacrificed (AP system).
  • S2 refuses to respond until the partition heals → availability sacrificed (CP system).

You cannot have both C and A during a partition.

Real-world example: During a major Meta outage in 2021, an internal BGP configuration service became unreachable. Even though the data centers were physically running, the CP consistency requirement of the configuration system prevented services from serving traffic because they could not verify correct state.

Google Spanner note: Spanner uses GPS and atomic clock (TrueTime) to achieve near-perfect external consistency at global scale. It pushes the practical limits of CAP by making partition windows so small (< 7ms) that the CA trade-off becomes negligible in practice.

Why this architecture?

Every distributed system designer must choose a CAP position for each data store explicitly, based on the business requirement:

  • Bank balance → CP (Postgres, Redis). A stale balance read can result in fraudulent overdraft.
  • Shopping cart → AP (DynamoDB). It is acceptable for the cart to show slightly stale data; it must always be available.
  • User profile → AP (Cassandra). Showing a 10-second stale bio is acceptable; a failed profile load is not.

What can fail?

Failure Mode CP System Behavior AP System Behavior
Network partition Rejects writes/reads to ensure consistency Serves stale data; reconciles after partition heals
Node failure Availability drops (fewer nodes to quorum) System remains fully available; stale data risk increases
Partition heals Resumes normal operation Reconciliation process runs; conflict resolution needed (last-write-wins, vector clocks)

Conflict resolution in AP systems (Cassandra, DynamoDB):

  • Last-Write-Wins (LWW) — The write with the latest timestamp wins. Risk: clock skew causes legitimate writes to be overwritten.
  • Vector Clocks — Each node tracks a version vector; the system detects and presents conflicting versions to the application for resolution (Amazon Dynamo's original design).
  • CRDTs (Conflict-free Replicated Data Types) — Mathematical structures that always merge correctly (e.g., a counter that only increments). Used by Riak and Redis CRDT module.

Measuring Availability

1. Time-Based Availability (using MTTF and MTTR):

Availability here means the share of time the service is usable. It is estimated with MTTF (Mean Time To Failure, the average healthy runtime before the next outage) and MTTR (Mean Time To Recovery, the average time to restore service after an outage).

Availability = MTTF / (MTTF + MTTR)
  • MTTF (Mean Time To Failure) — Average time between the end of one outage and the start of the next.
  • MTTR (Mean Time To Recovery) — Average time to detect, diagnose, and restore service.

Availability SLA (service level agreement, the external promise made to customers) "nines":

Availability Annual Downtime Example
99% ~3.65 days Acceptable for internal tooling
99.9% ~8.7 hours Most B2B SaaS products
99.99% ~52 minutes Amazon S3, major payment APIs
99.999% ~5 minutes Telecom, financial exchanges

2. Request Count-Based:

Availability = Successful Requests / Total Requests

Drawback: Biased toward power users (who generate 1000× more requests). A system down at 3am appears highly available because almost no requests are made.

3. Google's Meaningful Availability (User Uptime):

Availability = Σ uptime(u) / (Σ uptime(u) + Σ downtime(u))  for all users u

Counts every failed request (even silently retried ones) as contributing to downtime. The most conservative and user-centric metric. Recommended for user-facing products.

Techniques for Improving Availability

Redundancy: Deploy multiple copies of each subsystem. Stateless services + load balancer (front doors that spread traffic across multiple servers) = horizontal redundancy.

  • Without redundancy: 1 server + 1 DB → MTBF dominated by hardware MTBF (~5 years). Any hardware failure = outage.
  • With redundancy: 2 servers + primary/replica DB → single component failure is transparent to users. Availability improves from ~99.9% to ~99.99%.

Idempotency, Retries, and Deduplication

These three concepts always travel together in distributed systems and show up in nearly every design that involves network calls, queues, or payments.

Why retries are necessary: In a distributed system, any network call can fail with an ambiguous result — you do not know whether the server received the request and crashed before responding, or never received it at all. Retrying is the only way to guarantee eventual success. But naive retries cause duplicate processing.

Idempotency is the property that applying the same operation multiple times produces the same result as applying it once. Safe HTTP methods (GET, PUT, DELETE) are idempotent by definition; POST is not.

Making POST idempotent:

  • Clients attach an Idempotency-Key header (a UUID generated per logical operation).
  • The server stores (key → response) in a durable store (Redis with a TTL, or a DB table).
  • On retry, the server finds the existing key and returns the cached response without re-executing the operation.
  • Example: Stripe requires an Idempotency-Key on every charge creation. If a charge request times out, the client retries with the same key and Stripe returns the original charge result — the customer's card is never charged twice.

Deduplication on the consumer side: Message queues guarantee at-least-once delivery, not exactly-once. A consumer may process a message twice if it crashes after processing but before acknowledging. To prevent duplicate side effects:

  • Maintain a processed_ids set (in Redis or DB).
  • Before processing a message, check whether its ID is already in the set.
  • After processing successfully, insert the ID.
  • This converts at-least-once delivery into effectively-once behavior at the application layer.

Retry strategy:

  • Use exponential backoff with jitter: delay = min(cap, base * 2^attempt) + random(0, jitter).
  • Jitter prevents synchronized retry storms when many clients fail at the same time.
  • Set a maximum retry count and route failed messages to a DLQ after exhaustion.

Chapter 6: Database Internals

Problem Statement

Understand the storage-engine trade-offs behind Postgres, MySQL, RocksDB, and Cassandra.

Plain-English intuition

A storage engine is the part of the database that decides how bytes are laid out on disk and how reads, writes, recovery, and compaction (background merging that rewrites data into cleaner files) actually happen. The same logical row can be stored in a way that favors fast reads, fast writes, or cheap crash recovery, but usually not all three at once.

A B-tree (sorted tree structures used for indexes (extra data structures that make lookups faster) and range queries) behaves like a sorted library catalog: finding one book or a range of books is efficient because the shelves stay ordered. An LSM tree (write-optimized storage structures that merge data in the background) behaves more like an incoming-mail room: new updates are accepted quickly into a write-optimized buffer and organized later in the background. That makes writes cheap, but compaction and multi-level reads become the trade-off.

Real-world example:

Postgres usually powers transactional systems such as orders, payments, and user accounts, where indexed lookups and range scans must stay predictable. RocksDB and Cassandra are a better fit for write-heavy workloads such as clickstreams, IoT events, or log ingestion, where millions of writes per second matter more than perfect range-query performance.

What Actually Happens On Read And Write Paths

That ordering is the core durability (whether acknowledged data survives crashes) idea: the log becomes durable before the main data pages are lazily updated. Crash recovery replays the WAL to rebuild a consistent state.

MVCC (multi-version concurrency control, keeping multiple row versions so reads and writes interfere less) path

MVCC usually means writes create a new row version rather than overwriting the old one in place. Readers use a snapshot to decide which version is visible. That is why a 5-minute report can continue reading a consistent old view while checkout traffic keeps inserting and updating rows. The downside is cleanup work: old versions accumulate until vacuum or compaction removes them.

LSM path

An LSM engine does the opposite of a B-tree in a useful way:

  1. Append to WAL.
  2. Write to an in-memory memtable (the in-memory write buffer used by LSM-based databases).
  3. Flush immutable memtables to SSTables (immutable sorted files written by LSM-based databases) on disk.
  4. Compact overlapping SSTables into fewer larger files over time.

This makes writes fast and sequential, but creates read amplification and background compaction cost. Bloom filters (compact probabilistic structures that quickly say a key is probably absent), block caches, and leveled compaction exist largely to control those side effects.

B-Trees vs. LSM Trees

B-tree

  • Optimized for reads and range scans.
  • Keeps data and index in a balanced tree.
  • Good when read latency (how long one request or operation takes) matters and write volume is moderate.

LSM tree

  • Optimized for write-heavy workloads.
  • Appends to memory first, then compacts to disk later.
  • Used by RocksDB and Cassandra.

Trade-off:

  • B-tree: lower read amplification, higher random write cost.
  • LSM: lower write cost, higher compaction and read amplification.

WAL and Durability

Write-ahead logging means:

  1. Append the change to the WAL.
  2. Flush WAL to durable storage.
  3. Apply the change to the main data structure.

If the DB crashes, the WAL replays the committed writes.

Real-world example: A payment system should only tell the user "payment successful" after the WAL record is durable. Otherwise the service may acknowledge a charge, crash, and restart without any durable record of that charge.

MVCC

Multi-version concurrency control keeps multiple versions of a row so readers do not block writers.

Benefits:

  • Snapshot isolation.
  • Better read concurrency.
  • Cleaner transactional semantics.

Real-world example: An analyst can run a 5-minute sales report while checkout traffic is still creating new orders. The report reads a consistent snapshot, and the writes continue without blocking it.

Transaction Isolation Levels

MVCC enables multiple isolation levels. The SQL standard defines four; most production systems use one of the middle two.

Isolation Level Dirty Reads Non-Repeatable Reads Phantom Reads Description
READ UNCOMMITTED Possible Possible Possible A transaction can read uncommitted changes from other transactions. Almost never used.
READ COMMITTED Not possible Possible Possible A transaction only sees committed rows. The default in Postgres and Oracle. A re-read of the same row inside one transaction may return a different value if another transaction committed between reads.
REPEATABLE READ Not possible Not possible Possible (in theory) A transaction always sees the same snapshot of data it first read, even if other transactions commit. Postgres's implementation prevents phantoms too via MVCC snapshots.
SERIALIZABLE Not possible Not possible Not possible The strongest guarantee. Transactions behave as if they ran one at a time in some serial order. Postgres uses Serializable Snapshot Isolation (SSI), which adds conflict detection without full locking.

Practical guidance:

  • Default to READ COMMITTED for most OLTP workloads; it provides good performance and no dirty reads.
  • Use REPEATABLE READ when a single transaction performs several reads that must see a consistent snapshot (e.g., generating a report inside a transaction).
  • Use SERIALIZABLE only when the application truly needs serial correctness (e.g., "seat reservation must not double-book") — it adds overhead from conflict tracking.

Real-world example: Stripe's payment-processing transactions use SERIALIZABLE isolation for balance checks and charge application so that concurrent requests for the same user cannot both see "balance sufficient" and proceed.

Indexing Strategies

Indexes are extra data structures that make lookups faster, but they increase write cost and storage. Each write must update every index on that table.

Index Type What It Does When To Use
Primary (clustered) Orders the physical row storage by the index key. Only one per table. Integer or UUID primary key lookups.
Secondary (non-clustered) Separate structure pointing back to the heap. Alternate lookup paths: WHERE email = ?, WHERE status = 'active'.
Composite Index on multiple columns together. Queries filtering or sorting on multiple columns in a specific order. (user_id, created_at) covers WHERE user_id = ? ORDER BY created_at.
Covering All columns the query needs are in the index, so no heap lookup is required. Read-hot queries where eliminating the "table fetch" hop matters.
Partial Indexes only rows matching a condition. Selective predicates where most rows are irrelevant. WHERE is_active = true indexes only active users, keeping the index small.
Expression / Functional Indexes the result of an expression. LOWER(email) for case-insensitive lookups; DATE(created_at) for day-level aggregations.
Full-text GIN / GiST index on text with inverted structure. LIKE '%keyword%' alternatives; see Chapter 20 for search engine internals.
BRIN (Block Range) Stores min/max values per block of rows. Append-only time-series or log tables where rows are naturally ordered by time. Very small overhead.

The index decision rule: add an index only when a query is running against production data and EXPLAIN ANALYZE shows a sequential scan with high row cost. Every index you add increases INSERT/UPDATE/DELETE cost by approximately one B-tree write, plus extra storage (~30–50% of the indexed column size).

Index bloat: Postgres does not reuse dead index entries without a VACUUM. On tables with high update rates, run VACUUM ANALYZE regularly and monitor pg_stat_user_indexes for bloat.

Partitioning and Sharding

Partitioning and sharding both split data, but at different scopes.

Table partitioning (within one database): A single large table is divided into smaller child tables. The query planner knows which partition to touch for a given predicate, skipping others. Postgres 10+ supports declarative partitioning.

Partition Type How It Works Best For
Range Split by a column range, typically time (created_at < 2026-01-01). Time-series, logs, events. Simplifies TTL: drop old partition instead of DELETE.
List Split by discrete values (region = 'us-east'). Multi-tenant data, geographic splits.
Hash Split by hash of a key modulo N. Even distribution when no natural range or list boundary exists.

Horizontal sharding (across multiple databases): Shard key determines which database host owns a row. Queries must be routed to the right shard; cross-shard joins require application-level merging. Consistent hashing (see Chapter 13) is the standard routing mechanism.

Strategy Pros Cons
Range sharding Simple to reason about; sequential scans efficient Hot shards if recent data dominates
Hash sharding Even distribution Loses range locality; resharding is expensive
Directory-based Most flexible; shard assignment is a lookup Routing lookup adds a hop; directory becomes a hot path

Real-world example: Shopify shards its MySQL cluster by shop ID. Every query carries a shop context that resolves to the correct shard via a directory lookup. This lets them add capacity by adding new shards without migrating existing shops.

Connection Pooling

Each database connection is expensive: a Postgres connection costs ~5–10 MB of backend memory and requires a separate process/thread. At 10,000 concurrent application instances each holding one connection, the DB exhausts memory before it saturates CPU.

PgBouncer (Postgres connection pooler) sits between application and database and multiplexes thousands of application connections onto a much smaller number of actual server connections.

PgBouncer Mode How It Works Trade-off
Session mode Application connection is tied to one server connection for its lifetime. No multiplexing benefit; useful mainly for limiting max connections.
Transaction mode Server connection released back to the pool after each transaction. Most efficient; incompatible with some features (advisory locks, LISTEN/NOTIFY).
Statement mode Released after each statement. Rarely used; breaks multi-statement transactions.

Rule of thumb: Set max_pool_size to (CPU cores on DB node) × 2 + effective_spindle_count. For most cloud instances, 50–100 server-side connections is the practical ceiling. PgBouncer then allows thousands of client connections to land safely on those 100 backend connections.

Capacity Estimation

  1. Write-heavy workload: 50K writes/second with 1 KB rows is ~50 MB/s of raw ingest.
  2. WAL overhead: Plan for at least 1x additional write bandwidth for durable logging.
  3. LSM compaction: Expect multiple passes over the same data, so storage and I/O need headroom.
  4. Index cost: Every additional index increases write cost and storage footprint, so only add indexes that match real queries.

What Can Fail?

  • WAL disk fills up.
  • Compaction backlog increases tail latency.
  • Secondary indexes become stale or expensive.
  • Long-running transactions pin old versions and increase storage.

Why This Architecture?

The storage engine choice follows workload shape. A B-tree is the right default when the workload mixes reads and writes with range queries — most OLTP systems (orders, user accounts, payments) land here. An LSM engine is the right choice when write throughput far exceeds read throughput, such as time-series telemetry, clickstreams, and IoT sensor ingestion. There is no universally best engine; the question is always which of read amplification, write amplification, and space amplification (extra disk used beyond logical data size) the workload can afford.

A secondary dimension is durability guarantee. WAL-first systems (Postgres, MySQL InnoDB) trade some write throughput for crash safety. In-memory systems (Redis in AOF mode) may compromise on durability for speed. Choosing the wrong engine for a workload creates problems that are expensive to migrate away from later.

Common decision flow:

  • High write throughput (> 50 K writes/second), no complex joins needed → LSM (RocksDB, Cassandra, ScyllaDB).
  • ACID transactions, complex queries, moderate write throughput → B-tree (Postgres, MySQL).
  • Sub-millisecond reads, data fits in RAM, durability secondary → In-memory (Redis).
  • Very large dataset, mostly append, analytical scans → Columnar (Parquet on S3, BigQuery, Redshift).

How Will Teams Maintain It?

Database maintenance is not optional — a neglected database accumulates bloat, stale indexes, slow queries, and eventually fails in surprising ways.

Key practices:

  • Schema migrations — use Flyway, Liquibase, or Atlas. Never modify a production schema manually. All changes are version-controlled, run idempotently, and staged through staging before production. For zero-downtime migrations, follow the expand-contract pattern described in Chapter 4 under Schema Evolution.
  • Slow query log — enable it and alert on queries exceeding 200–500 ms. In Postgres, pg_stat_statements provides cumulative slow-query data without live overhead. Review it weekly and act on the top 5 offenders.
  • EXPLAIN ANALYZE — before adding an index, run EXPLAIN (ANALYZE, BUFFERS) on the slow query in a staging clone with production-representative data. Confirm the planner would use the index.

  • Regular restore drills — restore a production backup to a staging instance once per quarter and verify data integrity. An undrilled backup is not a backup.
  • Connection pool monitoring — watch pg_stat_activity for idle-in-transaction connections that hold locks. Set idle_in_transaction_session_timeout = 30s in Postgres to kill them automatically.
  • Index bloat analysis — run pgstatindex or the pg_bloat_check tool periodically. Rebuild bloated indexes with REINDEX CONCURRENTLY.
  • Compaction monitoring for LSM (RocksDB/Cassandra) — watch compaction queue depth. A growing compaction backlog indicates write throughput is exceeding compaction capacity; add nodes or reduce write rate.

Chapter 7: Rate Limiting

Problem Statement

Control how many requests a user, API key, IP, or tenant can issue over time.

Plain-English intuition

Rate limiting (controlling how fast a client may send requests) is a fairness and safety mechanism, not just an anti-abuse feature. It answers questions like: how many login attempts should one IP get per minute, how many SMS messages can one tenant send per second, and how do we keep one noisy customer from consuming everyone else's capacity?

A useful mental model is a stadium entry gate. Token bucket (an algorithm that allows short bursts while refilling credits over time) means people may enter in short bursts if enough tickets have accumulated. Leaky bucket (an algorithm that smooths traffic at a steady rate) means security lets people through at a fixed pace even if the queue outside grows quickly.

Real-world example:

Cloudflare uses rate limiting to absorb abusive traffic before it reaches origin servers. Stripe uses it to keep one API client from overwhelming payment infrastructure. Slack uses it to stop one workspace or bot from monopolizing shared backend capacity during message bursts or bad integrations.

API Design

Design Axes

  • Scalability: rate limiting must scale with active key cardinality and request volume.
  • Latency (how long one request or operation takes) vs throughput (how much work the system finishes per second): each check sits on the request path, so it must be fast even at very high throughput.
  • Availability (how often the system stays usable) vs consistency (whether different parts of the system see the same data at the same time): distributed counters often prefer availability and approximate enforcement over perfect global consistency.
  • Reliability and fault tolerance: local fallback buckets and sharded counters prevent the limiter from becoming a single point of failure.
  • Synchronous vs asynchronous: the allow/deny decision is synchronous; analytics and reconciliation are asynchronous.

Algorithms

  • Token bucket: Tokens refill over time; requests consume tokens. Good for bursty traffic such as a user refreshing a dashboard several times quickly.
  • Leaky bucket: Requests drain at a constant rate. Good for smoothing output to downstream systems such as SMS providers or email APIs.
  • Sliding window (counting requests inside a moving time range) log: Store timestamps for exact accuracy. Good for strict controls such as failed login attempts or OTP verification. Expensive at scale.
  • Sliding window counter: Approximate, lower memory than a full log. Good for high-volume API gateways (single entry points that route, protect, and observe APIs) where a close approximation is acceptable.

Distributed Rate Limiting

Common pattern:

  1. API server receives a request.
  2. It checks a shared counter store such as Redis.
  3. A Lua script (a small script often run inside Redis atomically) or atomic increment updates the counter.
  4. The request is allowed or rejected based on the updated state.

For very high scale, keep a local token bucket per server and periodically reconcile with a shared global budget.

Choosing The Right Limit Scope And Storage Model

Real rate limiting is not just "requests per minute." You have to decide three things precisely:

  1. Who is being limited? IP, user, API key, tenant, device, region, or some combination.
  2. Where is the limit enforced? CDN (content delivery network, edge servers that cache content near users) edge, API gateway, service layer, or downstream worker.
  3. What resource is protected? CPU, DB connections, SMS spend, login attempts, or third-party API quota.

Those choices change the design. Login endpoints often need tight per-IP and per-account windows to stop brute-force attacks. A public API usually needs per-token or per-tenant fairness. A notification system may need limits tied to dollars spent or vendor quotas, not just request count.

The storage model also matters:

  • Exact logs store timestamps and are best when correctness matters, such as failed-login tracking.
  • Counters are cheaper and good enough for coarse quotas.
  • Token buckets are best when burst allowance should be explicit.
  • Hierarchical quotas combine local and global enforcement, such as "100 req/s per user" and "10K req/s per tenant."

Finally, decide fail behavior explicitly. For a billing API, failing open may create financial risk. For a login endpoint under attack, failing closed is often correct. For a low-risk analytics endpoint, failing open may be acceptable to preserve availability. The limiter is part of the product contract, not just infrastructure glue.

Capacity Estimation

  1. Checks per second: If 100K users each make 10 requests/second, the limiter handles 1M checks/second.
  2. Storage: One counter per user/IP/tenant is small, but millions of active keys still require sharding (splitting data across many machines).
  3. Bandwidth: At 200 bytes per check, 1M checks/second is ~200 MB/s of counter traffic if every check is centralized.
  4. Server count: Shard counters across multiple Redis nodes (individual machines or service instances in a larger system) or use local admission control to reduce pressure.

What Can Fail?

  • Shared counter store becomes a bottleneck.
  • Clock skew breaks time-window logic.
  • Too-strict limits block legitimate bursts.
  • Too-loose limits fail to protect downstream systems.

-- ARGV[2]: window size in seconds
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local now = tonumber(redis.call("TIME")[1])  -- current unix timestamp

-- Remove events outside the sliding window
redis.call("ZREMRANGEBYSCORE", key, 0, now - window)

-- Count events in window
local count = redis.call("ZCARD", key)

if count < limit then
    -- Allow: add current timestamp to the sorted set
    redis.call("ZADD", key, now, now .. math.random())
    redis.call("EXPIRE", key, window)
    return {1, limit - count - 1}  -- allowed, remaining
else
    -- Reject: return time until oldest event exits the window
    local oldest = tonumber(redis.call("ZRANGE", key, 0, 0, "WITHSCORES")[2])
    local retry_after = window - (now - oldest)
    return {0, 0, retry_after}  -- denied, remaining=0, retry_after
end

This is called atomically via EVAL, ensuring there is no race between the check and the update — a common bug in naive multi-command implementations.

Token bucket in Redis — simpler for burst-tolerant limits:

FUNCTION allow_token_bucket(key, capacity, refill_rate_per_sec):
  tokens, last_refill = GET(key)
  now = current_time()
  # Refill tokens based on elapsed time
  tokens = min(capacity, tokens + refill_rate_per_sec * (now - last_refill))
  if tokens >= 1:
      SET(key, {tokens - 1, now})
      return ALLOW
  else:
      return DENY, retry_after = (1 - tokens) / refill_rate_per_sec

Why This Architecture?

Rate limiting exists because every resource — CPU, DB connections, third-party API quota, SMS spend, and outbound bandwidth — has a finite capacity that can be exhausted by a single noisy caller. Without rate limiting, one runaway client or one buggy retry loop can take down a shared service for all other users, and external API costs can spike by orders of magnitude overnight.

The algorithm choice is driven by the business requirement: token bucket for user-facing APIs where burst is acceptable (a user may legitimately fire 5 search queries in a second), leaky bucket when protecting a rate-limited downstream (a Twilio SMS API allows 100 outbound messages/second; excess must be smoothed, not bursted), and sliding window when precise enforcement is legally or contractually required (login attempts, OTP sends).

Fail behavior is a product decision, not an infrastructure default. For a critical billing API, failing closed (returning 429 and letting the client retry) prevents overcharging. For a non-critical analytics endpoint, failing open (allowing the request if the limiter is unreachable) preserves availability. Specify the fail behavior in the service's runbook.

Chapter 8: Distributed Consensus

Problem Statement

Keep a replicated state machine (machines applying the same ordered commands so they stay identical) consistent across failures.

Plain-English intuition

Consensus (a way for many machines to agree on one official answer) is what you use when "probably correct" is not acceptable. Several machines may each have partial information, but the system still needs one official answer to questions like "who is the leader?" or "was this configuration change committed?" Raft (a leader-based consensus algorithm designed to be easier to understand) solves that by requiring a majority to agree on the same ordered log.

Think about Kubernetes: when you scale a deployment from 3 pods to 5, etcd stores that desired state. If two control-plane nodes (individual machines or service instances in a larger system) disagree about the current value, the cluster can create duplicate or missing work. Consensus prevents that split view.

Real-world example:

ZooKeeper, etcd, CockroachDB, and Kafka KRaft (Kafka’s built-in metadata management based on the Raft algorithm) all rely on consensus-style leader election (choosing one node to coordinate writes or metadata) and replicated logs.

API Design

Design Axes

  • Scalability: consensus clusters usually stay small because quorum (the minimum number of replicas that must agree) coordination cost grows with node count.
  • Latency (how long one request or operation takes) vs throughput (how much work the system finishes per second): writes pay coordination latency, so throughput is lower than in leaderless systems.
  • Availability (how often the system stays usable) vs consistency (whether different parts of the system see the same data at the same time): consensus prioritizes consistency, which can reduce availability during partitions.
  • Reliability and fault tolerance: leader election, log replication (keeping copies of data in sync on multiple machines), and snapshots keep state recoverable.
  • Stateful vs stateless: every node is stateful because it stores logs and snapshots.

Raft In Practice

Raft is easier to explain than Paxos (a classic consensus algorithm for keeping replicas in agreement):

  • One leader at a time.
  • Leader appends log entries.
  • Followers replicate the log.
  • A majority quorum commits the entry.

Real-world example: If the leader appends feature_flag_x = true, the change is only committed after a majority acknowledges it. If the leader crashes before that majority exists, the cluster must treat the entry as uncommitted.

Raft Step By Step

Raft is easier to trust when you understand the steady-state loop and the failure loop.

Steady state

  1. One node is the leader for the current term.
  2. Clients send proposals to the leader.
  3. The leader appends the proposal to its log and sends AppendEntries RPCs to followers.
  4. Once a majority stores the entry, the leader marks it committed and applies it to the state machine.
  5. Followers apply committed entries in the same order.

Failure and election

  1. Followers expect regular heartbeats from the leader.
  2. If a follower times out, it increments the term and becomes a candidate.
  3. The candidate requests votes from peers.
  4. Peers vote only if the candidate's term is newer and its log is at least as up to date as theirs.
  5. If the candidate wins a majority, it becomes the new leader. If not, it falls back and retries later.

The safety property comes from these rules: only one leader can win a majority in one term, and leaders cannot overwrite committed entries from older majorities. Log matching via (prevLogIndex, prevLogTerm) ensures followers reject inconsistent histories until the leader backs up and repairs them. Snapshotting exists because replaying millions of old log entries on every restart would eventually become too slow.

Where It Shows Up

  • etcd — the Kubernetes control plane uses etcd for all cluster state (desired pod count, config maps, service endpoints). Every kubectl apply goes through etcd. Loss of etcd quorum halts the control plane.
  • ZooKeeper — historically used by Kafka for broker metadata, Hadoop for HDFS NameNode HA, and HBase for region server coordination. Being replaced by Raft-based alternatives in newer systems.
  • Kafka KRaft — Kafka's built-in metadata management mode, replacing ZooKeeper since Kafka 3.x. Uses the Raft algorithm inside the Kafka binary.
  • CockroachDB — uses Raft per-range so each partition of a globally distributed SQL table maintains its own consensus group independently.
  • Consul — service discovery and distributed KV store using Raft; also provides distributed locks and leader election via sessions.

Read Linearizability and Follower Reads

A subtle but important operational point: not all reads from a consensus cluster are automatically linearizable.

In Raft, followers can serve reads from their local state — but a follower might be slightly behind the leader. To guarantee a linearizable (most-recent-value) read, the system must either:

  1. Route all reads to the leader. Safe but creates a hot leader, reducing read throughput.
  2. Leader lease reads. The leader confirms it is still the active leader (via a recent heartbeat within a bounded time window) before serving the read locally. This avoids a quorum round trip but requires tight clock discipline.
  3. Quorum reads. Read from a quorum of nodes and return the value with the highest log index. Safe but doubles read latency.

etcd behavior: By default, etcd routes reads to the leader to guarantee linearizability. Clients can opt into serializable (follower) reads for lower latency at the cost of possible staleness.

CockroachDB behavior: Closed timestamps allow followers to serve consistent reads for past timestamps without leader involvement, enabling near-linear read scale-out.

For interviews: if you propose using etcd or Consul for configuration and coordination, be prepared to say whether your readers need linearizable reads or whether slightly stale reads are acceptable — the answer changes the architecture meaningfully.

Paxos In One Sentence

Paxos proves that a quorum can agree on a value even with failures, but the protocol is harder to reason about than Raft for interview purposes.

What Can Fail?

  • Leader crashes before commit.
  • Network partition (a failure where machines cannot communicate with each other) splits the quorum.
  • Log divergence after partial failures.
  • Slow followers lag far behind the leader.

Why This Architecture?

Consensus gives distributed systems a single agreed-upon source of truth for critical metadata: who is the leader, what is the current configuration, what log entries are committed. Without consensus, distributed systems fall back to "hope" — one node might act as leader while another, not knowing of the first, also acts as leader, creating split-brain and conflicting state (see Chapter 9 for split-brain details).

The alternative — using a single central node for coordination — trades consensus complexity for a simpler SPOF (single point of failure). That is the right trade-off for small systems. At large scale, the coordination node itself becomes the availability bottleneck, and a consensus cluster (3 or 5 nodes, tolerating 1 or 2 failures) is the robust replacement.

When to use consensus:

  • Leader election for database primaries, Kafka controllers, or scheduler masters.
  • Distributed locks and leases where correctness matters more than throughput.
  • Configuration management where all nodes must agree on the current configuration before acting on it.

When not to use consensus:

  • Data replication at high write throughput — consensus clusters stay small and cannot scale horizontally like primary-replica systems. Use Raft for metadata, not for the full user-data write path.
  • When eventual consistency is acceptable — leaderless replication (Cassandra, DynamoDB) handles high-throughput writes without a consensus layer.

Consensus clusters are intentionally small (3 or 5 nodes), so raw infrastructure cost is low. The more significant costs are latency and operational care.

Write latency cost: Every write that must commit through a quorum adds at least one network round trip (~1–5 ms within an AZ, ~10–30 ms cross-AZ). How Will Teams Maintain It?

Consensus clusters are critical infrastructure. A quorum loss halts the entire system that depends on it (e.g., all Kubernetes scheduling stops when etcd loses quorum). Treat these clusters like databases — with the same rigor around monitoring, backups, and runbooks.

Key operational practices:

  • Monitor quorum health continuously. Alert if any node is not in the quorum or if leader elections happen more than once per hour. Frequent elections indicate network instability or CPU starvation on cluster nodes.
  • Watch replication lag per follower. A follower that falls too far behind the leader is at risk of requiring a full snapshot transfer to catch up, which can take minutes and causes a brief loss of fault tolerance.
  • Monitor snapshot size and log retention. Unbounded log growth eventually causes disk exhaustion, which is a hard failure. Set compaction policies and alert at 70% disk usage.
  • Back up etcd regularly. etcdctl snapshot save produces a single-file backup. Run it hourly and store snapshots in S3. Test restore quarterly — an etcd restore is not trivial and should not be done for the first time during an incident.
  • Do not run etcd on shared hardware. Consensus performance depends on consistent low-latency disk I/O (for WAL writes). Running etcd on a node with noisy neighbors causes heartbeat timeouts and spurious elections. Use dedicated nodes with fast SSDs.
  • Stage changes through a single node at a time. When upgrading or replacing a node, remove one, verify quorum is healthy, then add the new node. Never replace multiple nodes simultaneously — you may lose quorum mid-operation.
  • Set alert thresholds for election churn. Raft leader election churn (more than one election per 10 minutes) indicates a problem with the leader (CPU, disk, or network) that should be investigated before it escalates to quorum loss.

Chapter 9: Distributed Systems

Problem Statement

This chapter covers the control-plane concepts that separate senior distributed-systems engineers from architects: leader election (choosing one node to coordinate writes or metadata), consensus (a way for many machines to agree on one official answer), replication (keeping copies of data in sync on multiple machines) models, distributed locks, clock synchronization, eventual consistency (replicas may disagree briefly but converge later), split brain, quorum (the minimum number of replicas that must agree) reads/writes, delivery guarantees, and saga (a multi-step workflow that uses local transactions plus compensating actions) orchestration.

These are the mechanics behind systems that use ZooKeeper, etcd, Raft (a leader-based consensus algorithm designed to be easier to understand), Paxos (a classic consensus algorithm for keeping replicas in agreement), and CAP-aware storage layers.

Example Workflow

Imagine a checkout flow spread across multiple services. Inventory reserves stock, Payments charges the card, Shipping creates a label, and Notification sends confirmation. This one user action touches leader election, replication, retries, idempotency (making duplicate processing safe), and saga compensation. These topics matter because they are the difference between "order created once" and "customer charged twice."

Real-world example:

These systems exist because one machine cannot safely coordinate a fleet by itself.

Why This Architecture?

Distributed systems (systems where many machines cooperate to act like one) need a way to make progress when nodes fail or networks partition. Consensus, quorum rules, and leader election let the cluster keep a single source of truth for metadata or critical writes. Without these mechanisms, systems drift into split brain and unsafe concurrent updates.

Replication Models

  • Primary-replica: one writer, many followers. Simple and low latency for reads, but failover (switching traffic or leadership after a failure) must promote a new leader.
  • Leaderless replication: any node can accept writes; replicas reconcile later. Improves availability but complicates conflict resolution.
  • Synchronous replication: write is acknowledged only after replicas confirm. Better consistency, worse latency.
  • Asynchronous replication: primary acknowledges immediately and followers catch up later. Better latency, higher risk of stale reads after failure.

Clock Synchronization

Distributed systems do not get perfect time, only bounded skew.

  • NTP is common for loose synchronization.
  • TrueTime-like APIs provide bounded uncertainty for globally consistent systems.
  • Leases and TTLs must account for skew or they can expire too early or too late.

Quorum, Leases, And Fencing Tokens

Quorum rules are easy to state and easy to misuse. If a replicated store has N replicas, a common pattern is to choose read quorum R and write quorum W such that:

text
R + W > N

The intuition is that every successful read overlaps with every successful write on at least one replica, which makes stale reads less likely. But quorum by itself is not enough for safety in leader-based coordination.

Consider a distributed lock. Node A acquires the lock, then gets partitioned from the rest of the system. Its local process may still believe it owns the lock even after the lease has expired elsewhere. If node B acquires the lock later, both A and B may try to write. This is where fencing tokens matter. Every lock or leadership grant carries a monotonically increasing token. Downstream systems reject writes with older tokens. That way, even if node A is stale and still "thinks" it is leader, it cannot perform valid writes after node B receives a newer token.

This is the subtle reason that "just use Redis for a distributed lock" is often incomplete. A lease without fencing protects liveness, but not necessarily correctness. For correctness-critical workflows, monotonic epochs or tokens are what stop stale leaders and zombie workers from corrupting state.

Event Delivery Guarantees

Three semantics define how many times a consumer processes each event:

Semantic Guarantee How achieved Use when
At-most-once Message delivered 0 or 1 times. No duplicates but possible loss. Commit offset before processing; if consumer crashes after commit but before processing, the message is skipped. Low-value metrics where loss is acceptable; logging pipelines where a gap is fine.
At-least-once Message delivered 1 or more times. No loss but possible duplicates. Commit offset only after processing completes successfully. If consumer crashes mid-processing, the broker re-delivers on restart. Requires idempotent consumers. Most production event systems. Kafka's default with manual offset commit.
Effectively-once (end-to-end) Appears exactly once from the application's perspective. At-least-once delivery + idempotent consumer-side deduplication (see idempotency keys in Chapter 5 and the outbox pattern in Chapter 4). True "exactly-once" at the broker layer (Kafka transactions) adds overhead and is rarely necessary. Payment processing, billing, order placement — anywhere duplicates cause visible damage.

Why "exactly-once" at the broker layer is tricky: Kafka provides exactly-once semantics within a single Kafka cluster using transactions and idempotent producers. This prevents duplicate writes to Kafka. But once the event leaves Kafka and triggers an action in an external system (send email, update DB, call payment API), exactly-once is no longer a Kafka problem — it requires idempotency on the external side. True end-to-end exactly-once requires coordination across Kafka, the consumer, and every external system it touches. Most production systems accept at-least-once delivery and invest in idempotency instead.

Commit position matters: In Kafka, consumer offsets are the bookmark into the log. If a consumer processes a batch of 100 messages and commits the offset after success, it achieves at-least-once. If it auto-commits before processing, it achieves at-most-once. Choosing commit strategy is one of the most consequential decisions in a Kafka consumer's design.

Saga Pattern

See Chapter 4 (Microservice Architecture Principles) → "The Saga Pattern: Distributed Transactions Without 2PC" for the full step-by-step walkthrough with examples, choreography vs. orchestration tradeoffs, and the compensating transaction pattern.

The summary relevant to this chapter's control-plane context: a Saga avoids distributed two-phase commit (2PC) by replacing one cross-service atomic transaction with a sequence of local transactions plus explicit compensating actions for rollback. Use a Saga when:

  1. A business workflow spans multiple services with separate databases.
  2. The business can tolerate eventual consistency — the state may be inconsistent for seconds or minutes while compensations propagate.
  3. 2PC's blocking behavior and coordinator dependency are unacceptable for the system's availability requirements.

The Saga coordination plane itself must be reliable. A Saga orchestrator is a stateful service — if it crashes mid-flow, it must be able to resume. Store Saga state durably (in Postgres or a Kafka compacted topic) and design the orchestrator to be restartable from any step.

Split-brain deep dive: When two nodes both believe they are the leader, the cluster produces conflicting writes. The standard mitigation is a fencing token — a monotonically increasing integer granted with each leader lease. Storage systems that receive writes check the token and reject any write with a token lower than the highest they have seen. This means a stale leader — one still running after its lease expired and a new leader was elected — will have its writes silently rejected, preventing data corruption. Fencing tokens are covered in detail in the "Quorum, Leases, And Fencing Tokens" section above.

What Can Fail?

  • Leader election flaps cause write unavailability.
  • Split brain creates two writers and conflicting state.
  • Clock skew breaks leases and TTL (Time To Live, meaning how long cached or temporary data stays valid)-based coordination.
  • Weak quorum settings can lose acknowledged writes.
  • Async replication can return stale data after a failover.
  • Saga compensation can fail and leave partial business state.

How Much Will It Cost?

  1. Consensus overhead: a quorum write costs extra network round trips and CPU.
  2. Replication overhead: every replicated byte costs bandwidth on each follower.
  3. Distributed locks: lock managers require highly available coordination infrastructure, often a small but critical cluster.
  4. Operational overhead: engineering time for observability, failover tests, and incident handling often dominates the raw compute cost.

How Will Teams Maintain It?

  • Monitor leader churn and election frequency.
  • Alert on quorum loss, replication lag, and stale followers.
  • Keep nodes time-synchronized and alert on skew.
  • Use idempotency keys for retry-safe writes and queue consumers.
  • Run failure drills for partitions, leader crashes, and stale lock cleanup.
  • Document every saga with explicit compensating actions and rollback behavior.

  • Latency vs throughput (how much work the system finishes per second): stronger consistency increases write latency and lowers throughput.

Chapter 10: Security Design

Problem Statement

Design the security layer for a modern distributed system (systems where many machines cooperate to act like one): authenticate users, authorize actions, protect data in transit and at rest, manage secrets, rate limit abuse, defend against DDoS, and reduce exposure to the OWASP Top 10.

Security is not a single component. It is a layered system that spans identity, transport, storage, runtime, and operational controls.

Example Security Flow

Consider an admin support tool for refunds. A support agent signs in with password plus MFA (multi-factor authentication, meaning more than one proof of identity), receives a short-lived access token, the API gateway (single entry points that route, protect, and observe APIs) verifies the token signature, the refund service checks the agent's role and region, secrets are fetched from a vault, the request is written to an audit log, and the database data remains encrypted at rest. Security is this whole chain, not just the login page.

Real-world example:

OAuth 2.0 (a delegated authorization framework) is used by many consumer platforms for delegated access, while JWT (JSON Web Token, a signed token that carries claims) is often used to carry signed identity claims between services. Cloudflare, AWS WAF, and API gateways sit in front of services to absorb attacks and enforce policies. Stripe and GitHub rely heavily on scoped authorization, auditability, and strict secret management because a security failure is a business failure.

Threat Modeling By Layer

Security gets clearer when you stop treating it as a checklist and instead walk layer by layer through the system.

  1. Identity layer — how users and services prove who they are. Threats: credential stuffing, session theft, token replay, weak MFA flows.
  2. Transport layer — how traffic moves between browser, edge, and services. Threats: plaintext hops, bad TLS (the encryption layer that protects data while it travels over the network) configs, certificate expiry, and man-in-the-middle exposure.
  3. Application layer — what authenticated actors are allowed to do. Threats: broken access control, IDOR, confused-deputy behavior, injection, XSS, and SSRF.
  4. Data layer — where sensitive data is stored and replicated. Threats: unencrypted backups, over-broad DB permissions, unsafe exports, and missing retention controls.
  5. Operations layer — how secrets, deploys, dependencies, and logs are handled. Threats: leaked API keys, vulnerable libraries, CI artifacts with credentials, and missing audit trails.

For each layer, ask four questions: what asset exists here, who could abuse it, what trust boundary protects it, and what control detects or blocks the abuse? That habit produces much better designs than memorizing product names like "use OAuth" or "use a WAF." Those are tools, not the threat model.

OAuth 2.0

OAuth 2.0 is an authorization framework, not an authentication system by itself. It delegates access decisions to a trusted Authorization Server rather than passing credentials directly between services.

Core roles in every OAuth 2.0 flow:

Role Responsibility
Resource Owner The user or entity that owns the data (e.g., the person whose calendar is being read)
Client The application requesting access on behalf of the resource owner
Authorization Server Issues tokens after authenticating the user and obtaining consent
Resource Server Hosts the protected API; validates the token on each request

Common flows:

  • Authorization Code Flow — most common for web and mobile apps. The user authenticates at the Authorization Server, which returns a short-lived authorization code. The client exchanges the code for tokens server-to-server, keeping tokens out of the browser URL bar. Use PKCE (Proof Key for Code Exchange) for SPAs and native apps to prevent code interception attacks.
  • Client Credentials Flow — no user involved. A backend service authenticates with its own client_id and client_secret to receive an access token. Used for service-to-service calls, cron jobs, and background workers.
  • Device Authorization Flow — for input-constrained devices (smart TVs, CLIs). The device shows a URL and a code; the user authenticates on a secondary device. The device polls until the user approves.
  • Refresh Tokens — long-lived tokens that obtain new access tokens without re-prompting the user. Must be stored securely (HttpOnly cookie or secure storage), rotated on use, and invalidated on logout.

Token types:

  • Access token — short-lived (typically 15 minutes to 1 hour). Sent as a Bearer token in the Authorization header. May be a JWT or an opaque reference token.
  • Refresh token — longer-lived (hours to days). Never sent to the Resource Server; used only with the Authorization Server to rotate access tokens.
  • ID token — a JWT from OpenID Connect (OIDC, the identity layer built on top of OAuth 2.0) that carries the user's identity claims. Not an access token — do not use it to call APIs.

Common pitfalls:

  • Returning tokens in redirect URLs rather than server-to-server exchange exposes them in browser history and logs.
  • Accepting tokens without validating the aud (audience) claim allows token substitution attacks across services.
  • Issuing long-lived access tokens without a revocation strategy means a stolen token stays valid until expiry.
  • Not using PKCE for public clients (SPAs, mobile apps) leaves the authorization code vulnerable to interception.

OAuth 2.0 is how a user can say, "This app may read my calendar," without handing over their password. OpenID Connect (OIDC) adds a standard identity layer on top so the app also knows who that user is.

JWT

JWTs (JSON Web Tokens) are compact, URL-safe tokens that carry signed claims between parties. They enable stateless verification: a service can confirm authenticity and read the payload without contacting a central token store.

Structure: Three Base64URL-encoded segments separated by dots: header.payload.signature.

eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9   ← header (algorithm + type)
.eyJzdWIiOiJ1c3JfMTIzIiwiaXNzIjoiYXV0... ← payload (claims)
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV...   ← signature

Standard claims (registered in RFC 7519):

Claim Meaning Example
sub Subject — who the token is about usr_123
iss Issuer — who minted the token https://auth.example.com
aud Audience — intended recipient https://api.example.com
exp Expiration time (Unix timestamp) 1745283600
iat Issued-at time 1745280000
jti JWT ID — unique token identifier for revocation tok_abc
scope or roles Authorization claims ["read:orders", "write:profile"]

Signing algorithms:

  • HS256 (HMAC-SHA256) — symmetric; same secret signs and verifies. Simpler but every verifier must hold the secret. Appropriate for single-service use.
  • RS256 (RSA-SHA256) — asymmetric; private key signs, public key verifies. Resource Servers only need the public key, which can be published via JWKS (JSON Web Key Set) endpoint. Preferred for multi-service architectures.
  • ES256 (ECDSA-SHA256) — smaller signature than RSA at equivalent security. Increasingly common for mobile-heavy systems where token size matters.

Validation checklist (perform all steps on every request):

  1. Verify the signature against the expected algorithm and key.
  2. Confirm iss matches the expected issuer.
  3. Confirm aud includes your service's identifier.
  4. Confirm exp is in the future and iat is not in the future.
  5. If revocation matters, check jti against a blocklist or use short expiry.

Stateless verification is a double-edged sword. Because JWTs are self-contained, a token that is leaked or stolen stays valid until exp. Mitigations:

  • Keep access token lifetime short (15 minutes is a common default).
  • Use refresh token rotation — each use of a refresh token issues a new one and invalidates the old one. Detect reuse attacks (an old token being used after rotation) by checking the token family.
  • Maintain a token blocklist in Redis keyed by jti for immediate revocation after logout or compromise. The performance cost is one Redis GET per request, which is acceptable at most scales.
  • Avoid placing sensitive PII in the payload — the payload is Base64-encoded, not encrypted. Anyone with the token can decode and read it. Use JWE (JSON Web Encryption) if confidentiality of claims is required.

JWT vs. opaque tokens:

Dimension JWT Opaque token
Verification Local (stateless) Remote (introspection endpoint)
Revocation Hard (must wait for expiry or use blocklist) Easy (delete from token store)
Token size Larger (carries claims) Small (random identifier)
Best for Distributed services needing stateless verification Scenarios requiring immediate revocation

RBAC vs ABAC

Authorization answers the question: "Is this authenticated identity allowed to perform this action on this resource?" The model you choose determines how expressive and how maintainable that answer is.

RBAC (Role-Based Access Control):

Permissions are assigned to roles; roles are assigned to users. A user's effective permissions are the union of all permissions across their roles.

User → [admin, billing_viewer] → {create_refund, view_invoice, ...}

Strengths: easy to reason about, audit-friendly, simple to implement, and maps naturally to enterprise org charts. Adding a new employee means assigning roles, not writing new policy.

Weaknesses: coarse-grained. A role can only capture "what" a user may do, not "when," "where," or "on whose resource." Role explosion — creating dozens of near-identical roles to capture edge cases — is a common failure mode.

Real-world examples: admin, support_agent, billing_viewer, read_only. Most SaaS products start here.

ABAC (Attribute-Based Access Control):

Access decisions evaluate a policy expression over attributes of the subject (user), resource, action, and environment.

ALLOW IF:
  user.department == resource.department
  AND user.clearance_level >= resource.sensitivity
  AND request.time BETWEEN 09:00 AND 17:00
  AND request.region == "us-east-1"

Strengths: fine-grained and contextual. A single policy can replace dozens of RBAC roles. Supports dynamic decisions based on runtime context (IP, time of day, device trust level).

Weaknesses: harder to reason about, audit, and debug. Policy engines (OPA — Open Policy Agent, Cedar, Casbin) add operational complexity. A policy bug can be subtle and affect many users at once.

Real-world examples: multi-tenant SaaS where a user may only edit documents in their own organization; healthcare systems where a doctor may only access records for patients under active care; financial platforms where fraud analysts may only view transactions flagged in their assigned region.

Choosing between them:

Scenario Recommendation
Small team, simple roles, enterprise B2B RBAC
Multi-tenant with per-tenant permissions RBAC + tenant scoping
Fine-grained row-level or field-level access ABAC or hybrid
Contextual constraints (time, location, device) ABAC
Regulatory compliance requiring rich audit trails ABAC with policy-as-code

Many production systems use both: RBAC for coarse role assignment and ABAC expressions for fine-grained resource ownership checks. For example, role=support_agent gates entry to the support tool (RBAC), while ticket.assigned_team == agent.team gates access to individual tickets (ABAC).

Policy-as-code: For complex ABAC systems, use OPA (Open Policy Agent) or AWS Cedar to define, test, and version authorization policy as code. Policies are reviewed in pull requests, run in CI against example inputs, and deployed independently of application code. This prevents authorization regressions from being invisible.

Encryption

Encryption is a foundational control, but it is scoped: it protects data from specific threats (eavesdropping, stolen storage media) and does not replace access control, input validation, or secure application logic.

Encryption in transit:

All traffic should be encrypted with TLS 1.2 or higher. TLS 1.3 is preferred: it removes weak cipher suites, reduces handshake latency (one fewer round trip), and mandates forward secrecy.

Key practices:

  • Enforce HTTPS for all client-facing endpoints. Redirect HTTP to HTTPS and use HSTS (HTTP Strict Transport Security) with a long max-age to prevent downgrade attacks.
  • Encrypt internal service-to-service traffic, not just the public edge. A compromised internal network should not expose plaintext traffic. Use mTLS (mutual TLS) to authenticate both sides of internal calls.
  • Rotate TLS certificates before expiry using automated tools (Let's Encrypt with cert-manager, AWS ACM). Certificate expiry is the most common cause of preventable TLS outages.
  • Validate server certificates in all clients — SDK and internal service clients should not silently accept self-signed certificates in production.
  • Disable TLS 1.0 and 1.1 at the load balancer. These versions have known weaknesses (BEAST, POODLE).

Encryption at rest:

Data on disk should be encrypted so that stolen storage media or cloud snapshots reveal nothing useful.

Layer What to encrypt Typical tool
Database storage All rows and indexes AWS RDS encryption (AES-256), Postgres with LUKS
Object storage Files, exports, backups S3 SSE-S3 or SSE-KMS; enforce via bucket policy
Backups DB dumps, WAL archives Encrypt before upload; store keys separately
Application-level PII fields (SSN, card number) Envelope encryption via KMS
Logs Audit logs containing sensitive context Encrypt at rest + restrict access

Envelope encryption is the industry-standard pattern for database columns holding sensitive values:

  1. A data encryption key (DEK) encrypts the plaintext field value.
  2. The DEK itself is encrypted by a master key stored in a KMS (Key Management Service, such as AWS KMS or HashiCorp Vault).
  3. Only the encrypted DEK is stored next to the data. The plaintext DEK lives in memory only during active use.

This means rotating the master key does not require re-encrypting all data — only the DEKs need to be re-wrapped, which is fast.

Key management principles:

  • Never store encryption keys and the data they protect in the same location.
  • Use hardware-backed key stores (HSM, AWS KMS, GCP Cloud HSM) for master keys.
  • Rotate keys on a defined schedule (annually for at-rest keys, more frequently for signing keys).
  • Log all key usage to an append-only audit trail.
  • Document and test the key rotation runbook before it is needed in an incident.

What encryption does not protect against:

  • A bug in the application that returns decrypted data to the wrong user (broken access control).
  • SQL injection that queries and returns encrypted-then-decrypted fields through normal application logic.
  • An attacker who compromises the application process itself, which holds the decryption keys in memory.
  • Insider threat where an authorized user with database access exports data legitimately.

Encryption reduces the blast radius of storage theft and snapshot leakage. It is a necessary layer but never a sufficient one.

Secrets Management

Store and rotate:

  • API keys
  • database passwords
  • signing keys
  • TLS certificates
  • webhook secrets

Use managed secret stores or HSM-backed systems when possible. Never hardcode secrets in source code or bake them permanently into images.

Rate Limiting and DDoS Protection

Rate limiting belongs here because security and abuse prevention are tightly connected.

  • Use per-user, per-IP, and per-tenant limits.
  • Combine token bucket (an algorithm that allows short bursts while refilling credits over time) or sliding window (counting requests inside a moving time range) logic with edge enforcement.
  • Put Cloudflare, WAF rules, and load balancers (front doors that spread traffic across multiple servers) in front of origin services.
  • Use bot detection and anomaly detection for volumetric and application-layer attacks.

What Can Fail?

  • Password or token theft.
  • JWT signing key leakage.
  • Authorization bug leading to privilege escalation.
  • TLS misconfiguration or expired certificate.
  • Secret exposure in logs, code, or CI artifacts.
  • Rate limiting too weak to stop abusive traffic.
  • DDoS overwhelms the edge or origin.
  • Injection or XSS vulnerability in a user-facing flow.
  • Token revocation or logout not propagating quickly enough.

How Much Will It Cost?

  1. TLS: certificate automation is cheap; the main cost is engineering and testing.
  2. Secrets management: managed vaults, KMS, and HSM-backed keys add recurring cost but reduce breach risk.
  3. WAF / DDoS protection: edge security products can be a meaningful monthly line item, especially at high traffic volumes.
  4. Authorization and audit logs: more logging and policy checks increase storage and compute usage.
  5. Security reviews and pen tests: the operational cost is often higher than the infra cost.

Security is one of the highest ROI investments because one breach can dwarf many months of prevention spending.

How Will Teams Maintain It?

  • Rotate keys and certificates on a schedule.
  • Use least privilege for services, users, and operators.
  • Review and test authorization policies regularly.
  • Log auth events, policy decisions, and security-sensitive actions.
  • Run dependency scanning and patch vulnerable libraries quickly.
  • Keep secrets out of source control and CI logs.
  • Use SAST, DAST, and dependency audits in CI/CD.
  • Perform tabletop incident drills for credential leakage and account takeover.

Examples: /orders/{id} returns any order regardless of who owns it (IDOR — Insecure Direct Object Reference); a regular user accesses admin endpoints. Enforce authorization on every request server-side. Use ACL or ABAC checks, not just UI hiding. Return 403, not 404, to avoid leaking existence. Test with automated tooling (Burp Suite, custom fuzz tests).

Examples: Log4Shell (CVE-2021-44228), Spring4Shell, an outdated OpenSSL version with a critical TLS flaw. Run automated dependency scanning in CI (Dependabot, Snyk, OWASP Dependency-Check). Examples: weak or missing brute-force protection on login, sessions not invalidated on logout, predictable session tokens, credential stuffing. Rate-limit login attempts (see Chapter 7). A08 Software and Data Integrity Failures Code or data updates applied without integrity verification. Examples: CI/CD pipeline fetching dependencies without checksum verification (supply chain attack), deserialization of untrusted objects leading to RCE, unsigned firmware updates. A09 Security Logging and Monitoring Failures Insufficient visibility into attacks. Examples: login failures not logged, no alert on 100 failed auth attempts per minute, audit logs not tamper-proof, no tracing on privilege escalation events. Where OWASP failures show up in distributed systems specifically:

  • Injection (A03) surfaces most often in search, filtering, and reporting endpoints that construct queries dynamically.
  • SSRF (A10) is especially dangerous in microservices where a single service can reach internal APIs that trust traffic from within the VPC.
  • Broken access control (A01) in multi-tenant systems often manifests as missing tenant-scope filters — WHERE tenant_id = ? omitted from one query returns all tenants' data.
  • Security misconfiguration (A05) is the most common finding in cloud-native systems: overly permissive IAM roles, open security groups, and unencrypted S3 buckets.
  • Secret leakage: API keys appear in logs or repos. Mitigation: secret scanning and immediate rotation.
  • DDoS flood: bot traffic overwhelms the edge or the login endpoint. Mitigation: WAF, rate limits, CAPTCHA/challenges, and CDN (content delivery network, edge servers that cache content near users) shielding.

Chapter 11: Caching Strategies

Problem Statement

Design the caching layer for a modern distributed system (systems where many machines cooperate to act like one) so that the database is protected from repeated reads, latency (how long one request or operation takes) stays low, and hot traffic can be served efficiently.

This chapter covers cache-aside (a cache pattern where the app loads data into cache only after a miss), write-through (a cache pattern where writes update cache and database together), write-back (a cache pattern where writes hit cache first and reach durable storage later), TTL (Time To Live, meaning how long cached or temporary data stays valid) strategy, distributed caching, cache invalidation, hot keys (popular keys that receive far more traffic than average), and session caching.

Example Request Flow

A user opens an e-commerce product page. The web tier first checks Redis for product:123. On a hit, the page returns in a few milliseconds. On a miss, the service reads Postgres, returns the response, and writes the result into cache for the next shopper. If price or stock changes, the product service invalidates or refreshes that cache entry so the next read is fresh.

Real-world example:

Redis is commonly used for session storage, counters, and hot objects. Memcached is often used for simple, high-throughput (how much work the system finishes per second), in-memory caching of database results. Large internet systems like Facebook and Twitter rely on caching layers because serving every request from the database would be too slow and too expensive.

Why This Architecture?

Caching exists because memory is much faster than disk and because databases are too expensive to hit for every request. A good cache turns repeated reads into fast lookups and turns a hot database into a much smaller backend concern.

Cache Patterns

  • Cache-aside: application owns the cache fill logic. This is the most common pattern for product pages, blog posts, and profile reads.
  • Write-through: cache and database are written together. Good for sessions, feature flags, and user preferences where reads should reflect writes quickly.
  • Write-back: writes land in cache first and are flushed to storage later. Good for throughput-heavy counters or buffered analytics, but harder to recover from failure.

TTL Strategy

TTL is the simplest invalidation tool.

  • Short TTLs reduce staleness but increase DB load.
  • Long TTLs reduce DB load but risk stale reads.
  • Jitter TTLs to avoid simultaneous expiry and cache stampedes.
  • Use different TTLs for different data classes: product catalog, user sessions, feature flags, and rate-limit counters should not all share the same policy.

Distributed Caching

Distributed caches shard keys across nodes using consistent hashing (routing keys so only a small subset moves when nodes change) or a similar routing strategy.

Common technologies:

  • Redis — supports rich data types, persistence options, replication (keeping copies of data in sync on multiple machines), and clustering.
  • Memcached — simple, fast, in-memory key-value cache with a smaller feature set.

Cache Invalidation

Invalidation is hard because stale data is easy to create and hard to detect.

Common strategies:

  • Explicit delete on write
  • Write-through update
  • TTL expiry as a safety net
  • Versioned keys, where a new version naturally invalidates the old entry

How To Reason About Freshness

Cache freshness is not binary. Different data types need different guarantees:

  • Read-after-write consistency — when a user updates their own profile photo, they expect to see the new version immediately.
  • Bounded staleness — a product catalog can often tolerate being 30 seconds old.
  • Stale-while-revalidate (serving slightly old data while refreshing it in the background) — a response may be briefly stale while the system refreshes it in the background, which is often a great trade-off for read-heavy traffic.
  • Negative caching (temporarily caching "not found" results) — even "not found" responses may be cached briefly to avoid repeated misses hammering the database.

The right design depends on how writes happen. If writes originate in one service, explicit invalidation or versioned keys work well. If writes can happen in multiple places, change-data-capture or event-driven invalidation becomes safer. If freshness is extremely important, you may skip caching entirely on that path or use short TTL plus write-through updates.

A useful rule is to define freshness per dataset, not per cache cluster. Sessions, feature flags, prices, search results, and marketing pages should not all share one default TTL. Caching becomes predictable only when freshness expectations are part of the data model.

Hot Keys

Hot keys are accessed disproportionately often, such as:

  • homepage data
  • trending items
  • celebrity profiles
  • global feature flags

Hot keys can overload one cache node even if the average load looks fine. Solutions include replication, local read replicas (extra copies of data on other machines), request coalescing, and key splitting.

Session Caching

Session caching stores login state, auth tokens, or user context in a cache to avoid repeated database hits.

Typical uses:

  • web sessions
  • refresh-token lookup
  • MFA (multi-factor authentication, meaning more than one proof of identity) challenge state
  • user-specific feature flags

What Can Fail?

  • Cache stampede (many requests missing the same popular key at once) when many clients miss the same key at once.
  • Stale data after updates or invalidation delay.
  • Hot key overload on one cache shard.
  • Cache node failure causes a spike in DB traffic.
  • Write-back data loss if dirty cache contents are not flushed.
  • Session expiration logging users out unexpectedly.

How Much Will It Cost?

  1. Memory: cache is paid for in RAM, which is more expensive than disk.
  2. Replication: distributed caches often need replicas, which multiply memory cost.
  3. Network: remote cache access adds network hops and bandwidth cost.
  4. Operational overhead: cache tuning, invalidation logic, and monitoring take engineering time.

Redis often costs more than storing the same data on disk, but it is usually far cheaper than scaling the database to serve every request.

How Will Teams Maintain It?

  • Monitor cache hit rate, miss rate, eviction rate, and memory usage.
  • Watch for hot keys and uneven shard distribution.
  • Warm caches after deploys or failovers.
  • Use TTLs and invalidation policies documented per dataset.
  • Test failover (switching traffic or leadership after a failure) and restore paths.
  • Keep cache key naming conventions consistent across teams.

  • Scalability: caches scale horizontally because a single node is not enough at large data sizes.
  • Latency vs throughput: cache hits optimize latency and protect database throughput.
  • Availability (how often the system stays usable) vs consistency: caches often prefer availability and speed over perfectly fresh reads.
  • Reliability and fault tolerance: replication, fallback to DB, and pre-warming reduce outage impact.
  • Stateless vs stateful: cache nodes are stateful, while app servers are usually stateless.
  • Synchronous vs asynchronous: reads are synchronous; warming, invalidation, and backfill are often asynchronous.

Capacity Estimation

  1. Hit rate target: if 90% of 1M requests/second should hit cache, the cache tier must serve ~900K requests/second.
  2. Object size: 1 KB objects at 900K QPS is ~900 MB/s of read bandwidth across the fleet.
  3. Memory footprint: 100 GB logical cache with 2 replicas requires ~200 GB of raw cache memory.
  4. Shard count: if one node safely holds 50 GB, a 100 GB logical cache needs at least 2 primaries, plus replicas and spare headroom.

Real-world failure modes

  • Cache stampede: many clients miss the same popular key. Mitigation: TTL jitter, request coalescing, or mutex locks.
  • Hot key meltdown: one shard is overloaded by a single key. Mitigation: split the key, replicate reads, or move to a separate cache tier.
  • Invalidation lag: stale content persists after update. Mitigation: versioned keys or write-through writes.
  • Session loss: a cache outage logs users out. Mitigation: replicate sessions or store refresh tokens in a durable store.

  1. When is cache-aside the right default, and when is write-through worth the extra write-path complexity?
  2. How do you choose between TTL-based freshness and explicit invalidation for a dataset like prices, profiles, or feature flags?
  3. If the cache is unavailable, when should the system serve stale data, fall back to the database, or fail fast?

Chapter 12: Messaging & Event-Driven Architecture

Problem Statement

Design the messaging and event-driven backbone of a scalable system so that producers and consumers are decoupled, workloads can be buffered, and failures do not cascade through synchronous call chains.

This chapter covers queues, pub/sub (publish-subscribe, where one event can go to many consumers), event sourcing (storing every state change as an event log), CQRS (separating write models from read models), retry mechanisms, dead letter queues (holding areas for messages that keep failing), idempotency (making duplicate processing safe), and backpressure (slowing incoming work when downstream systems are overloaded).

Example Event Flow

A shopper places an order. The Order Service writes one order_created event. Payment consumes it to charge the card, Inventory consumes it to reserve stock, Email consumes it to send confirmation, and Analytics consumes it to update dashboards. None of those downstream actions need to block the original checkout request, which is why event-driven design is so powerful.

Real-world example:

Apache Kafka powers high-throughput (how much work the system finishes per second) event pipelines at many large companies. RabbitMQ is common for task queues and routing-heavy workloads. Amazon SQS is a managed queue often used for simple asynchronous workflows. These systems let one part of the system publish work while another part consumes it later.

Why This Architecture?

Event-driven systems decouple producers from consumers. That gives you better resilience, easier scaling, and the ability to replay or audit work later. It also allows different consumers to react to the same business event independently.

Queues vs Pub/Sub

  • Queues: one message goes to one consumer instance. Best for task distribution such as image resizing or invoice PDF generation.
  • Pub/Sub: one event can fan out to many consumers. Best for domain events such as order_created, where billing, email, analytics, and fraud systems all react independently.

Event Sourcing and CQRS

  • Event sourcing: the event log is the source of truth. State is rebuilt by replaying events. A bank ledger is the classic example: balance is derived from credits and debits.
  • CQRS: writes and reads are handled by different models. The write side stores commands/events; the read side stores optimized query projections. A ride-hailing app may store trip events on the write side and a denormalized "recent trips" view on the read side.

Retry Mechanisms

Retries should be designed with caution:

  • use exponential backoff
  • add jitter
  • cap retry count
  • avoid retry storms
  • distinguish transient from permanent failures

Retries improve reliability, but unbounded retries can amplify failures.

Dead Letter Queue

The DLQ holds messages that repeatedly fail. Common causes:

  • malformed payload
  • schema (the structure and rules of stored data) mismatch
  • poison pill
  • downstream outage

DLQs are operational safety valves, not a substitute for fixing the root cause.

Idempotency

At-least-once delivery is safe only if the consumer is idempotent. Common techniques:

  • idempotency keys
  • deduplication tables
  • exactly-once side effects at the business layer

Backpressure

Backpressure prevents slow consumers from being overwhelmed.

Mechanisms:

  • bounded queues
  • producer throttling
  • consumer scaling
  • shedding low-priority work

Ordering, Replay, And Consumer State

Most event-driven designs look simple until you ask what happens after failure and replay.

Ordering

Ordering is usually guaranteed only within one partition, not across the whole topic. That means if per-user or per-order ordering matters, partition by that key. If you partition randomly for better load distribution, you usually give up strict per-entity order.

Replay

Replay is valuable only if consumers are designed for it. A consumer should be able to reprocess old events without creating duplicate side effects. That often means:

  • commit offsets only after the side effect succeeds,
  • use idempotency keys or deduplication tables,
  • separate pure state-projection consumers from effectful consumers like billing or email.

Consumer state

Some consumers are stateless handlers. Others maintain materialized views, local RocksDB state, or compacted-topic snapshots. Those consumers need their own recovery story: snapshot restore, changelog replay, schema compatibility, and backfill procedures.

This is also why the outbox pattern (writing events in the same DB transaction as the business change, then publishing them later) matters. If a service writes to its database and publishes to the broker (the system that stores and routes messages) in separate steps, one may succeed while the other fails. Writing the event into the same local transaction (groups of operations that succeed or fail together) as the business row, then publishing from an outbox relay, is one of the cleanest ways to avoid that split between data and events.

What Can Fail?

  • Broker outage or partition loss.
  • Consumer lag grows without bound.
  • Poison pill message blocks processing.
  • Duplicate delivery causes double charges or duplicate notifications.
  • Schema drift breaks deserialization.
  • Backpressure is ignored and memory usage spikes.

How Much Will It Cost?

  1. Broker storage: retained messages consume disk and replication (keeping copies of data in sync on multiple machines) multiplies the footprint.
  2. Consumer fleets: more consumers are needed as throughput increases or processing becomes heavier.
  3. Replay storage: event sourcing and audits require longer retention than simple task queues.
  4. Operational overhead: schema management, monitoring, replay tooling, and incident response add cost.

Kafka often costs more than a simple queue, but it enables replay, fan-out (sending one event to many downstream targets), and high throughput at scale.

How Will Teams Maintain It?

  • Monitor consumer lag, broker disk, and partition skew.
  • Version schemas and use a schema registry.
  • Define DLQ handling and replay runbooks.
  • Make consumers idempotent by default.
  • Load test retries and failure scenarios.
  • Separate critical and non-critical traffic with priority queues or topics.

Chapter 13: Consistent Hashing

Problem Statement

Map keys to a changing set of servers while minimizing remapping when nodes (individual machines or service instances in a larger system) are added or removed.

This is the routing primitive behind Chapters 4, 7, and 8. The core challenge: distributed systems constantly gain nodes (capacity additions, recovery) and lose nodes (failures, decommissions). Any routing scheme that forces mass key movement when the fleet changes will collapse under load.

Plain-English Intuition

Imagine a clock face numbered 0–360. Each server is assigned a random position on the clock by hashing its name or IP. Each key is also placed on the clock by hashing it. To find which server owns a key, walk clockwise from the key's position until you hit a server. That server owns the key.

Now add a new server. It lands somewhere on the clock. Only keys that fall between the new server and its predecessor now change owners — every other key is unaffected. Similarly, removing a server only moves the keys it owned to its clockwise neighbor. This is why consistent hashing is called "consistent": the fraction of keys that move is proportional to 1/N, not (N-1)/N as with modulo hashing.

With normal modulo hashing, adding one cache node can move almost every key:

text
# Modulo hashing — catastrophic on resize
server = hash(key) % N          # N=8: key "foo" → server 3
server = hash(key) % (N+1)      # N=9: key "foo" → server 7  ← moved!

With consistent hashing, adding one node moves only the keys it "absorbs" from its clockwise neighbor. The rest of the fleet keeps serving hot data normally. This is why large systems like Memcached, Cassandra, and Dynamo-like stores can add capacity during a traffic spike without effectively cold-starting the whole cluster.

Real-World Examples

Amazon DynamoDB

DynamoDB uses a variant of consistent hashing called token-based partitioning. Each partition is assigned a token range on a 128-bit ring derived from the MD5 hash of the partition key. When a partition becomes too large (>10 GB) or too hot (>3,000 WCU), it is split: the token range is bisected and the two halves are assigned to different storage nodes. Only the keys in the affected range move. The rest of the ring is undisturbed. DynamoDB's control plane manages ring state via a distributed metadata service (similar to a Paxos-replicated log) so every router has a consistent view of which node owns which token range.

Apache Cassandra

Cassandra uses consistent hashing with vnodes since version 1.2. Every node in the cluster owns 256 virtual token positions by default (num_tokens=256 in cassandra.yaml). The ring state — which physical node owns which tokens — is gossiped around the cluster using the Gossip protocol (see "Protocols" section below). Clients use a native driver that maintains a local copy of the ring and routes directly to the owning replica without a proxy hop. When a node is decommissioned, Cassandra streams the leaving node's token ranges to its successors before updating the ring state, ensuring no data loss.

Memcached (libketama)

Before consistent hashing, Memcached clients used hash(key) % N. Adding one server invalidated ~87% of cached keys in a 7-server pool (6/7). LinkedIn's open-source libketama library (2007) introduced consistent hashing to Memcached clients. It builds a ring of 150 × N positions (150 vnodes per server), hashed using MD5. This reduced key movement on resize to roughly 1/N, preserving cache hit rate during scaling events.

Discord (Message Routing)

Discord routes WebSocket sessions across a fleet of gateway servers. Each user connection is assigned to a gateway server using consistent hashing on user_id. When a gateway server is restarted for a deploy, only the users assigned to that server's ring segment are disconnected and must reconnect — the rest stay connected. Without consistent hashing, a rolling restart would trigger mass reconnection storms.

Nginx and HAProxy (Load Balancing)

Both Nginx (hash $request_uri consistent) and HAProxy (balance uri) support consistent hashing for upstream selection. This is used to implement sticky routing for caches: all requests for the same URL hash to the same upstream cache server, maximizing cache hit rate while still tolerating upstream churn.

Why Consistent Hashing

Modulo hashing breaks badly when N changes:

text
server = hash(key) % N

If N changes from 8 to 9, most keys move. Consistent hashing keeps movement local: a key only moves when one of the nearby ring positions changes.

Key property (formally): If the hash function is uniform, adding a node to an N-node ring moves exactly K/N keys on average, where K is the total number of keys. Removing a node moves K/N keys. This is the theoretical minimum — you cannot do better without knowing the keys in advance.

Virtual Nodes (Vnodes)

Each physical server owns many positions on the ring, not just one.

Why: With only one position per server, the ring coverage is uneven by chance. Some servers own large arcs; others own small ones. This creates hot spots. With 200 vnodes per server, the law of large numbers kicks in and coverage becomes uniform.

Benefits:

  • Smooths out random hash imbalance.
  • Lets a single heavy server own proportionally more ring space.
  • Makes small fleets (3 nodes) behave statistically like large fleets.
  • Reduces the variance in how much data each node holds.

Weighted vnodes: If server has weight w_i, assign:

text
vnodes_i = round(total_vnodes * w_i / sum(all weights))

Example — heterogeneous fleet:

Node RAM vnodes (out of 600 total) Ring share
node-A 64 GB 150 25%
node-B 64 GB 150 25%
node-C 128 GB 300 50%

Node C owns twice the ring space because it has twice the capacity. Without weighting, it would be overloaded.

Overhead: 600 vnode entries × 16 bytes per token × 100 nodes = ~960 KB of ring metadata. This fits comfortably in memory and is cheap to gossip.

Ring Mechanics In Practice

In production, a hash ring is more than a diagram. Five operational details matter:

1. Lookup Algorithm

python
import bisect, hashlib

def get_node(key: str, ring: list[tuple[int, str]]) -> str:
    """
    ring: sorted list of (token, node_id) pairs
    Returns the node_id responsible for 'key'.
    """
    h = int(hashlib.md5(key.encode()).hexdigest(), 16)
    idx = bisect.bisect(ring, (h,))   # O(log V)
    idx = idx % len(ring)             # wrap around
    return ring[idx][1]

For replication, walk clockwise collecting the next R-1 distinct physical nodes after the primary.

2. Replication Strategy

Most systems replicate to the next R clockwise distinct physical nodes. Cassandra calls these the coordinator (primary) and replicas. For R=3:

Ring: ... A(token=10) → B(token=20) → C(token=30) → D(token=40) ...
Key hashes to token=15 → primary owner = B
Replicas: C (next distinct), D (next distinct after C)

This means any 1-of-3 replica can answer a read (with quorum reads, any 2-of-3 must agree).

3. Membership Propagation

Clients or routers need a current view of the ring. This is distributed through one of three mechanisms:

Mechanism How it works Used by
Gossip Each node periodically exchanges ring state with a random peer. Convergence in O(log N) rounds. Cassandra, Riak
Config service Ring state stored in a CP system (ZooKeeper, etcd, Consul). Clients watch for changes. Kafka (ZooKeeper era), Envoy/xDS
Client-side ring Clients download full ring state on startup and refresh on version mismatch. libketama, Redis Cluster

4. Safe Node Addition

1. New node joins, announces itself (gossip or registration).
2. Ring controller assigns token ranges (vnodes).
3. New node streams data from successor nodes for its assigned ranges.
4. Once streaming completes, ring state is updated to make new node live.
5. Predecessor nodes can drop their now-migrated data on next compaction.

This is a streaming before handoff pattern. It ensures data availability is never interrupted.

5. Safe Node Removal (Decommission)

1. Operator marks node as decommissioning.
2. Node streams all its token ranges to successor nodes.
3. Successor nodes confirm receipt and replicate to their own replicas.
4. Ring state updated to remove the decommissioning node.
5. Node shuts down cleanly.

Never abruptly remove a node without a drain. Abrupt removal can cause data loss if the node was the only replica for some token range.

Hash Functions in Practice

The hash function used to place keys and nodes on the ring matters significantly:

Function Speed Distribution Quality Collision Risk Used In
MD5 ~500 MB/s Excellent (uniform) Cryptographic, overkill libketama, old Cassandra
MurmurHash3 ~3 GB/s Excellent Non-crypto, fast Cassandra 3+, Redis, Guava
xxHash ~10 GB/s Excellent Non-crypto, fastest ClickHouse, modern systems

SHA-256 ~150 MB/s Excellent Cryptographic Security-sensitive contexts

Recommendation: Use MurmurHash3 or xxHash for consistent hashing. MD5 and SHA are much slower and the cryptographic security adds no value here since the ring is not security-sensitive. Avoid CRC32 for general use — it has known distribution weaknesses for certain input patterns.

Critical rule: All clients and servers in the cluster must use the same hash function and same seed. A mismatch means different clients compute different ring positions and route keys to wrong nodes.

Protocols and Standards

Gossip Protocol (Epidemic Protocol)

Used by Cassandra, Riak, and Serf to propagate ring membership changes:

  • Every T seconds (typically 1s), each node picks k random peers and exchanges state.
  • State includes: node list, their tokens, their health status, and a heartbeat counter.
  • Nodes merge incoming state with local state (higher heartbeat wins).
  • Dead detection: if a node's heartbeat has not advanced for phi intervals, it is suspected dead. After timeout, it is marked DOWN and its token ranges are covered by replicas.

Convergence: In a cluster of N nodes, gossip reaches all nodes in O(log N) rounds. For a 1000-node cluster, full convergence takes roughly 10 gossip rounds = ~10 seconds.

Chord Protocol (Academic Standard)

The original consistent hashing paper (Karger et al., 1997, MIT) defined the ring concept. The Chord DHT (Stoica et al., 2001, MIT) formalized it with:

  • A 160-bit ring (using SHA-1).
  • Finger table: each node stores pointers to log N successors at exponentially increasing distances, enabling O(log N) lookups across the ring without a central router.
  • Stabilization algorithm: a background process that periodically corrects finger tables as nodes join and leave.

While modern systems do not implement Chord exactly, its vocabulary (successor, predecessor, finger table, stabilize) is the standard terminology.

Consistent Hashing with Bounded Loads (Google, 2017)

Google published a refinement that adds a load bound: if any node's load exceeds the average by more than a factor ε, that node is skipped and the key is assigned to the next node clockwise. This prevents hot spots when traffic is skewed toward certain key ranges, at the cost of slightly breaking the "nearest clockwise node" invariant.

Used in: Google's internal load balancers, and the open-source Envoy proxy (as ring_hash and maglev policies).

Rendezvous Hashing (Highest Random Weight, HRW)

An alternative to ring hashing. For each key, compute hash(key, server_i) for every server and pick the server with the highest score.

text
owner = argmax_i( hash(key || server_i) )

Properties:

  • No ring data structure needed.
  • Perfectly uniform with any number of servers.
  • O(N) lookup per key (must score all servers).
  • On node removal, only the keys owned by that node move.

Used in: Nginx hash $uri, some CDN origin selection, Akamai routing.

Jump Consistent Hash (Google, 2014)

For N buckets, JumpHash produces a uniform assignment using only integer arithmetic — no ring, no vnodes, no sorted array:

c
int32_t JumpConsistentHash(uint64_t key, int32_t num_buckets) {
    int64_t b = -1, j = 0;
    while (j < num_buckets) {
        b = j;
        key = key * 2862933555777941757ULL + 1;
        j = (b + 1) * ((double)(1LL << 31) / (double)((key >> 33) + 1));
    }
    return (int32_t)b;
}

Properties:

  • O(log N) time, O(1) space.
  • Only supports adding nodes at the end (no arbitrary node removal).
  • Ideal for storage backends where nodes are added in sequence.

Used in: Google Spanner (internal variant), some distributed file systems.

Pattern 1 — Client-Side Ring (Smart Client)

Pro: No proxy hop; lowest latency.
Con: Ring state must be kept fresh on each client; client library complexity.

Pattern 2 — Proxy-Side Ring (Thin Client)

┌─────────────┐          ┌─────────────┐          ┌─────────────┐
│   Client    │──────────▶   Proxy /   │──────────▶│  Node A     │
│  (dumb)     │          │  Router     │          │  Node B     │
└─────────────┘          └─────────────┘          └─────────────┘

A central proxy (Twemproxy, mcrouter, Envoy) holds the ring and routes requests. Clients connect to the proxy using standard Memcached or Redis protocol.

Pro: Simple clients; ring state managed centrally.
Con: Proxy is a bottleneck and a single point of failure (must be replicated).

Pattern 3 — Sloppy Quorum with Hinted Handoff

Used by Amazon Dynamo and Cassandra for high availability:

  1. A write for key K targets replicas A, B, C (per the ring).
  2. If B is temporarily unavailable, write a hint to node D (outside the normal replica set), recording that this data belongs to B.
  3. When B recovers, D forwards the hinted write to B and discards the hint.

This allows writes to succeed even when designated replicas are down, at the cost of brief inconsistency. The sloppy quorum is W + R > N computed across the available nodes, not necessarily the designated ring nodes.

Pattern 4 — Token Range Splitting (Auto-Sharding)

Used by DynamoDB and TiKV:

  1. Monitor each partition's size and request rate.
  2. When a partition exceeds a threshold (e.g., 10 GB or 3,000 RCU), split its token range at the median key.
  3. Assign the two halves to different physical nodes.
  4. Update ring state atomically.

This allows the system to rebalance hot partitions automatically without operator intervention.

Pattern 5 — Virtual Shard (Two-Level Hashing)

Used by Redis Cluster:

  1. The ring has a fixed 16,384 slots (not vnodes). Slots never change.
  2. Each key maps to a slot: slot = CRC16(key) % 16384.
  3. Slots are assigned to physical nodes. Moving a node = moving its slot assignments.

The fixed slot count means clients cache a simple 16,384-entry slot→node table. Ring resharding just moves slot assignments, not vnodes. Clients can detect misroutes via MOVED and ASK redirections.

Load Distribution Math

With n servers and uniformly random keys:

  • Expected share per server: 1/n.
  • Variance with V vnodes per server: σ² ≈ 1/(n·V). More vnodes → less variance.
  • Standard deviation of load with 200 vnodes: σ ≈ 1/(n · √200) ≈ 0.07/n — within ±7% of mean.

Rule of thumb:

Vnodes per server Load imbalance (σ) Comment
1 High (~30%) Do not use
10 ~10% Borderline
100 ~3% Acceptable
200 ~2% Standard (Cassandra default)
1000 ~1% Diminishing returns, more metadata

If one server has 2x CPU/RAM, give it roughly 2x vnodes.

A 64 GB 300 25%
B 64 GB 300 25%

Capacity Estimation

  1. Key movement on resize: When one node joins or leaves a 16-node ring, ~1/16 = 6.25% of keys migrate.
  2. Rebalance bandwidth: Cluster stores 2 TB across 16 nodes (~125 GB each). Adding one node triggers ~125 GB of streaming. At 1 Gbps network, this takes ~17 minutes. Plan for this window.
  3. Metadata size: 200 vnodes × 100 nodes = 20,000 ring entries. At 20 bytes each = 400 KB. Trivial to cache on every client.
  4. Latency impact: Binary search on 20,000 sorted vnodes: ~14 comparisons = negligible vs. any network RTT.
  5. Replication amplification: With replication factor R=3, each write is sent to 3 nodes. Write throughput is effectively total_cluster_write_capacity / 3.

Why This Architecture?

Consistent hashing reduces churn when the fleet changes, which keeps scaling and failover (switching traffic or leadership after a failure) manageable. It is the reason distributed systems can scale horizontally without requiring a "freeze the cluster" window for rebalancing.

Without consistent hashing:

  • Adding a node to a cache pool invalidates most cache entries → thundering herd on the database.
  • Removing a node from a database cluster requires moving every key → hours of downtime or extremely complex migration tooling.

With consistent hashing:

  • Adding a node moves ~1/N of keys.
  • Removing a node moves ~1/N of keys.
  • The system stays live during both operations.

Interview Trade-Off Questions

  1. When is simple hashing good enough? If your node count never changes (fixed cluster, no autoscaling), modulo hashing is simpler and cheaper. Consistent hashing is necessary when the fleet changes under load.
  2. How many vnodes are enough? 150–256 is the standard range. More vnodes tighten the load distribution but increase metadata size, ring management complexity, and streaming parallelism during rebalancing.
  3. Rebalance during peak traffic — fast recovery or minimal movement? Prioritize minimal movement. A rebalance storm during peak traffic is far worse than a slightly longer recovery window. Throttle streaming bandwidth and defer non-critical rebalancing to off-peak windows.
  4. Vnodes vs. fixed slots (Redis Cluster style)? Vnodes give flexibility (any number of nodes, weighted). Fixed slots give simplicity (client routing table is a fixed array). For caches and key-value stores accessed by many different language clients, fixed slots are easier to implement correctly across all client libraries.
  5. What happens during a network partition that splits the gossip graph? Nodes on each side of the partition will have diverging ring views. To handle this safely: use a strongly consistent store (etcd, ZooKeeper) as the source of truth for ring state, and treat gossip as a performance optimization for fast propagation only.

Chapter 14: Designing a Distributed Cache

Problem Statement

Design a distributed in-memory cache that stores key-value pairs and absorbs the read load that would otherwise hit a database.

Plain-English Intuition

As a web service scales, the database becomes the bottleneck. Applications follow a Zipfian distribution — 80% of requests touch the same 20% of data. A cache stores that hot 20% in fast memory, serving it at ~100 ns instead of ~10 ms from disk. The challenge is keeping the cache consistent, distributed, and fault-tolerant.

Real-World Example

Facebook's Memcached cluster handles billions of cache reads per second, absorbing load that would otherwise hit MySQL shards. Without it, Facebook would need 100× more database servers. Amazon DynamoDB, Apache Cassandra, and the Memcached client library all use consistent hashing to distribute cache keys across nodes.

Functional Requirements

  1. put(key, value, ttl) — Store a key-value pair with optional expiry.
  2. get(key) — Return the value; null if not present or expired.
  3. delete(key) — Explicitly evict a key.
  4. Eviction: Automatically evict entries when memory is full, using a configurable policy (LRU, LFU, TTL).
  5. Replication: Data survives a single node failure without loss.
  6. Discovery: Clients automatically learn about node additions and removals without restart.

Non-Functional Requirements

Property Target

Throughput 1 M+ operations/second per cluster

Consistency Stale reads acceptable for short TTL windows
Scalability Horizontally scalable by adding nodes

  2. Storage: A 78 GB cluster stores ~78 M one-KB values before overhead. Replication factor 3 triples the physical footprint.
  3. Bandwidth: 800 K gets at 1 KB each = ~800 MB/s of read egress across the cluster; shard count and NIC capacity matter.
  4. Node count: 6 shards × 2 replicas = 12 nodes is a practical starting point for ~624 GB of logical capacity with failure tolerance.

Local Cache Design

A local LRU cache is the building block of all distributed caches.

Data Structures:

  • Hash Table — O(1) key lookup; returns a pointer to the node in the linked list.
  • Doubly Linked List — O(1) insert/delete. Head = most recently used; Tail = least recently used.

Retrieval (Get):

  1. Look up key in hash table. Miss → return null.
  2. Hit → move node to head → return value.

Insert/Update (Put):

  1. Key exists → update value, move to head.
  2. Key is new, cache not full → insert at head.
  3. Key is new, cache full → evict tail node (LRU), remove from hash table, insert new node at head.

Eviction Policies:

Policy Mechanism Best For
LRU Evicts item not accessed longest General-purpose; temporal locality
LFU Evicts item accessed least frequently Stable hot-set (e.g., celebrity profiles)
FIFO Evicts oldest-inserted item Simple use cases; no access locality
TTL-based Evicts item after a fixed time Session stores, rate-limit counters

Primary-Replica Replication:

  • Primary accepts writes; read replicas accept reads and asynchronously replicate from primary.
  • Hot shard → add more read replicas to that shard.
  • On primary failure, ZooKeeper triggers leader election; a replica is promoted. Reads continue uninterrupted throughout.

Design Axes

  • Scalability: Caches scale horizontally by sharding keys and adding replicas.
  • Latency vs throughput: Cache hits optimize latency while also protecting backend throughput.
  • Availability vs consistency: Caches favor availability; stale reads are tolerated for the TTL window.
  • Reliability and fault tolerance: Replication, failover, and cache warming keep the cache usable after failures.
  • Stateless vs stateful: Cache nodes are stateful; the application tier is stateless.
  • Synchronous vs asynchronous: Client reads/writes are synchronous; invalidation and warming are asynchronous.

Core Design Choices

  • Use consistent hashing with vnodes to route keys; avoids large-scale remapping on cluster changes.
  • Prefer a remote dedicated cache cluster over co-located caching for independent scaling and failure isolation.
  • Default to LRU eviction; switch to LFU for workloads with a stable hot set.
  • Use ZooKeeper for automatic node discovery so clients never require restarts on topology changes.
  • Replicate each shard to at least one replica in another availability zone for durability.

A single Redis node holds ~100 GB; a large product catalog or session store can reach terabytes, so sharding is unavoidable. Consistent hashing minimizes remapping disruption when nodes change.

Chapter 15: Designing an Auto-Complete Engine

Problem Statement

Design a search auto-complete engine that returns prefix-matched suggestions as a user types, ranked by popularity.

Example: User types "Dub" → system suggests: Dubai, Dublin, Dubrovnik.

Plain-English Intuition

Every keystroke fires a query. At 1,000 users typing simultaneously, that is thousands of sub-10 ms lookups per second. Disk-based lookups (~10 ms per seek) and database queries (20–100 ms) cannot meet that budget. The core trick is storing the entire dataset as an in-memory trie — a tree where each path from root to leaf spells a string. Prefix search becomes a single tree traversal: O(M) where M is the length of the typed prefix, not a function of dataset size.

Real-World Example

Google's suggest infrastructure serves billions of auto-complete queries per day with median latency under 5 ms. Airbnb and Amazon use the same architecture for destination and product search. The trie is rebuilt nightly from query logs; popularity scores are updated in near-real-time from a streaming pipeline.

Functional Requirements

  1. Suggest: Given a prefix string and locale, return the top-N completions ranked by popularity.
  2. Prefix match only: No substring, fuzzy, or semantic matching (those are handled by a separate layer).
  3. Popularity ranking: Results ordered by a popularity score computed from historical query frequency.
  4. Multi-language: Support Unicode prefixes and locale-specific ranking.
  5. Freshness: Popularity scores updated at least every 15 minutes from query logs.

Non-Functional Requirements

Property Target
Query latency (p99) < 10 ms
Indexed strings 10 billion
Peak throughput 1,000 QPS (scales to 10 K+ with caching)
Availability 99.99%+
Staleness Popularity scores up to 15 minutes old is acceptable

Capacity Estimation

  1. Storage: 10 B strings × 100 bytes/string = 1 TB raw data. Trie pointer overhead ~2× → ~2 TB effective memory required.
  2. Server count: 2 TB / 256 GB per node = ~8 nodes for storage; replication factor 2 + spare capacity → 16 nodes in practice.
  3. Query load: 1,000 QPS peak × avg 4 keystrokes per session = ~4,000 trie lookups/second; most are cache-hit after the first keystroke.
  4. Bandwidth: At ~1 KB per response, 1,000 QPS = ~1 MB/s egress — negligible.
  5. Popularity pipeline: Query logs at 1,000 QPS × 100 bytes/log = ~100 KB/s; easily handled by a batch job every 15 minutes.

Chapter 16: Designing a Scalable Notification Service

Problem Statement

Design a notification service that delivers push notifications, emails, SMS, and webhooks to millions of users within seconds of a triggering event, without coupling the upstream business service to delivery latency.

Canonical workflow: Driver cancels a ride → Ride Reservation Service emits an event → Notification Service → APNs/FCM → rider's phone within 5 seconds.

Plain-English Intuition

A naive design calls the Notification Service synchronously from the business service. If APNs is slow, the ride booking slows down too. The fix is to decouple: the business service records the event and returns immediately, while the notification system drains work asynchronously.

Another subtle point is that "send a notification" is actually several steps:

  • decide whether the user should receive it,
  • pick the allowed channels,
  • fan out to the right providers,
  • retry or suppress duplicates,
  • and record delivery state.

That is why real notification systems look like orchestration pipelines rather than simple sender services.

Real-World Example

Uber, DoorDash, and Stripe all use async notification pipelines. Notification services appear in virtually every product domain:

Domain Example Trigger Channel
Ride-hailing Driver cancelled ride Push + SMS
Finance Fraud alert Push + Email
E-commerce Item back in stock Email + Push
Social New follower In-app + Push
DevOps Service down PagerDuty (webhook)

Functional Requirements

  1. Send notification: Accept an event with recipient, channels (push/email/SMS/webhook), and payload; deliver asynchronously.
  2. Multi-channel delivery: Support APNs (iOS), FCM (Android/Web), email (SES/SendGrid), SMS (Twilio), webhooks, and in-app WebSocket delivery.
  3. Priority: Critical alerts (fraud, cancellation) must be delivered ahead of promotional notifications.
  4. Idempotency: Duplicate event submissions must not cause duplicate deliveries.
  5. Dead-letter handling: Messages failing after N retries land in a DLQ for inspection, not lost.
  6. User preferences: Respect opt-outs, quiet hours, and per-channel settings.
  7. Delivery receipts: Track whether each notification attempt was sent, delivered, opened, or failed.

Chapter 17: Lessons from Netflix's Zuul Push Service

Problem Statement

Design a large-scale real-time push platform that maintains millions of persistent device connections and routes targeted messages to the correct connection server with low latency and controlled operational blast radius.

Netflix built Zuul Push to deliver real-time updates to 250M+ subscriber devices across mobile, TV, and browser clients. The hard part is not just sending messages. It is keeping millions of long-lived connections healthy, deployable, and recoverable.

Plain-English Intuition

This system is different from a classic request/response API. The expensive resource is not CPU per request. It is open socket count. Once a client connects, the server must remember which machine owns that live connection so later messages can be routed to it.

That is why a push platform usually splits into three concerns:

  • a connection tier that owns the live sockets,
  • a registry that maps client_id -> server_id,
  • and a routing tier that consumes business events and forwards them to the right connection server.

When people say "WebSockets are fast," what they really mean is "the system pays the connection-management cost once, then pushes many messages over the same channel."

Real-World Example

When a Netflix session starts on one device, another device on the same account can receive a playback-state update almost immediately. Similar patterns show up in collaborative editors, multiplayer games, brokerage dashboards, and chat systems where low-latency fan-out matters more than raw request throughput.

Why WebSockets?

HTTP Long Polling Server holds connection until event Medium — holds resources open Better
Server-Sent Events (SSE) Persistent, server-to-client only Low Low
WebSockets ✅ Persistent, bidirectional Lowest (async, non-blocking) Lowest

WebSocket provides full-duplex TCP (a reliable transport protocol for moving data between machines) communication. Once connected, the server can push messages to the client at any time without the client asking.

Functional Requirements

  1. Register connection: Accept and authenticate long-lived client connections.
  2. Route message to client: Deliver a message to a specific connected device or session.
  3. Per-device fan-out: Support one user owning many devices with distinct active connections.
  4. Reconnection: Clients must recover automatically after routine failures or deploys.
  5. Best-effort ordering: Preserve message order per connection where practical.
  6. Presence awareness: Know whether a client is online and which server currently owns the connection.
  7. Fallback delivery: If the client is offline, optionally hand off to APNs/FCM or another async channel.

Chapter 18: Top-K Problem

Problem Statement

Find the Top K most frequent items over a given time window from a stream of billions of events.

API: topK(K, startTime, endTime) → [item_1, item_2, ..., item_K]

Plain-English Intuition

The difficulty is not computing Top K once. A single machine can count frequencies and keep a min-heap. The real problem is doing it continuously while events arrive at massive scale and while product requirements disagree about speed versus exactness.

Trending lists need answers in seconds, so approximate results are often acceptable. Billing, creator dashboards, and audit workflows need exact answers, so sketches are not enough. That naturally leads to a two-path design: a fast approximate path and a slower exact path.

Real-World Example

  • Top 100 trending photos on Instagram in the last 24 hours.
  • Most retweeted tweets on Twitter in the past hour.
  • Most played songs on Spotify today.
  • Most watched videos on YouTube this week.
  • Most searched queries on Google in the past minute.

Functional Requirements

  1. Top-K query: Return the most frequent items for a time window and scope.
  2. Windowing: Support windows such as last 5 minutes, 1 hour, 24 hours, or 7 days.
  3. Scope filters: Support global, regional, or category-specific Top K.
  4. Approximate fast path: Return near-real-time results in seconds.
  5. Exact historical path: Return exact results for analytics and audit use cases.
  6. Replay and backfill: Recompute windows when logic changes or processors fail.
  7. Hot-key handling: Avoid one viral item overwhelming a single partition.

Core Design Choices

Why not a single server with a hash table?
A single hash table could hold many counts, but ingestion rate becomes the bottleneck. Millions of events per second force horizontal partitioning for both compute and network.

Why partition by item key?
If events for the same item land on different hosts, no host knows the true count. Exact counting becomes much harder unless events are co-located by key.

Why two paths (fast + slow)?

  • Fast path: approximate, low-memory, seconds-level freshness.
  • Slow path: exact, replayable, and suitable for billing or auditing.

Count-Min Sketch

A hash table for frequencies grows unbounded with unique items. Count-Min Sketch provides approximate frequency counting in fixed memory.

Structure: a 2D array, H rows × W columns, where H is the number of hash functions and W is the width.

Update (seen item x):

For each row i (0 to H-1):
    sketch[i][h_i(x)] += 1

Query (estimate frequency of x):

Return min(sketch[i][h_i(x)]) for i in 0..H-1

Taking the minimum across rows minimizes over-counting from hash collisions.

Trade-off: Larger width and height reduce error but increase memory. The error bound is ±ε × N with probability 1 - δ.

Query Flow (Critical Path)

1. Client requests Top K for window W and scope S
2. Query API checks serving store for latest materialized result
3. If exact result exists, return it

Chapter 19: URL Shortener / Pastebin

Problem Statement

Create a service that turns long URLs or text blobs into short, durable, shareable identifiers.

Plain-English Intuition

A URL shortener has two very different workloads. The write path is "create a short code once." The read path is "redirect millions of clicks as fast as possible." That asymmetry is why the architecture usually keeps the redirect lookup extremely small and fast, while analytics, moderation, and cleanup happen asynchronously.

Pastebin adds another twist: the blob can be much larger than a URL. In practice, metadata such as code, owner, TTL (Time To Live, meaning how long cached or temporary data stays valid), and abuse status often lives in a database, while the raw paste body can live in object storage when snippets become large.

Real-World Example

TinyURL and bit.ly care deeply about redirect latency (how long one request or operation takes) because every shared marketing link, SMS campaign, or social post click passes through them. Pastebin-like systems care more about immutable storage, expiration, and abuse control because users often share logs, stack traces, or code snippets that may need moderation or short-lived retention.

Functional Requirements

  1. Shorten URL: Given a long URL, return a unique short code (e.g. sho.rt/aZ81kQ).
  2. Redirect: A GET on the short code resolves to the original URL with an HTTP redirect.
  3. Custom aliases: Users may optionally supply a vanity slug (e.g. sho.rt/my-product).
  4. TTL / expiration: Links can carry an expiry; after expiry the short code returns 410 Gone.
  5. Paste support: Accept arbitrary text blobs; return a code that serves the raw content.
  6. Delete link: Owners can deactivate a link before expiry.
  7. Analytics: Track redirect count, unique visitors, geography, referrer, and device type per link.
  8. Rate limiting: Prevent abuse on the creation endpoint.

Non-Functional Requirements

Durability No link lost once confirmed

Code uniqueness No two live codes may point to different targets

Capacity Estimation

  1. Code space: Base62 with 7 characters gives 62^7 ≈ 3.5 trillion possibilities, enough for billions of URLs without collision pressure.
  2. Storage: 100 M short links/day at ~100 bytes metadata each ≈ 10 GB/day of metadata.
  3. Redirect QPS: At 10 B redirects/day that is ~115 K redirects/second peak; the read path requires aggressive caching.
  4. Paste storage: Even at 100 KB average paste size, 1 M pastes/day = 100 GB/day; object storage is necessary at scale.
  5. TTL cleanup: A background sweeper or lazy-delete on access keeps expired rows from accumulating indefinitely.

Core Design Choices

  • Use a unique code generated by base62 encoding of a distributed counter or a random hash.
  • Redirect with 302 by default so every click is captured for analytics; use 301 only when analytics are unneeded and CDN caching is critical.
  • Store creation timestamp, owner, TTL, and redirect type in the Link table; keep analytics counts in a separate materialized table.
  • Keep the redirect path read-optimized: cache code → target in Redis and at the CDN edge; update analytics asynchronously via Kafka.
  • For Pastebin, store small blobs inline in the DB and large blobs in object storage; separate metadata from content so lookups stay fast.

Code Generation Deep-Dive

Two strategies are common:

  1. Counter-based (Base62 encode a distributed counter)

    • A dedicated counter service (Redis INCR or a Snowflake-style ID generator) issues monotonically increasing integers.
    • Encode to base62: integer 12345678 → "5afx".
    • Pros: zero collision, sortable, predictable length growth.
    • Cons: sequential codes are enumerable — anyone can crawl all links. Mitigate with a shuffle or random prefix.

  2. Hash-based

    • Hash the long URL (MD5/SHA-256), take the first N base62 characters.
    • Pros: same URL always produces the same code (idempotent creation).
    • Cons: collision risk (small but non-zero); must check DB and retry on collision.

Recommendation: use a counter for general links; use a CSPRNG (cryptographically secure random number generator) for private or sensitive links.

Chapter 20: Search Engine Design

Problem Statement

Design a search engine that crawls documents, builds an index (extra data structures that make lookups faster), ranks results, and serves queries with low latency (how long one request or operation takes).

This chapter extends the trie (a tree data structure optimized for prefix matching) and inverted-index ideas from Chapter 15.

Plain-English Intuition

Search is really two systems glued together: an offline pipeline that continuously discovers and organizes content, and an online serving path that answers user queries in milliseconds. The offline side can afford heavy computation. The online side cannot. That separation is the reason search engines feel fast even though indexing the web is enormous work.

A simple example: if a document contains the phrase redis consistency tradeoffs, the indexer stores entries such as redis -> doc42, consistency -> doc42, and tradeoffs -> doc42. At query time, searching redis consistency becomes an intersection of postings lists, not a scan of every document in the corpus.

Real-World Example

Google Search, Bing, Elasticsearch, and OpenSearch all use versions of this pipeline: crawl, index, rank, and serve.

Functional Requirements

  1. Crawl documents: Discover and fetch documents from configured seeds or domains.
  2. Parse and normalize: Extract text, metadata, links, canonical URLs, and language.
  3. Build inverted index: Map normalized terms to postings lists of matching documents.
  4. Serve search queries: Return ranked results within a low-latency budget.
  5. Snippets and highlights: Show short context snippets around matches.
  6. Incremental indexing: Reflect content changes without rebuilding the full corpus.
  7. Deduplication and canonicalization: Avoid indexing the same content under many URLs.
  8. Spam and quality filtering: Suppress low-quality or manipulated content.

Non-Functional Requirements

Property Target
Query latency (p99) < 200 ms
Query availability 99.99%
Crawl freshness Hot domains refreshed within minutes to hours
Durability Indexed data and crawl logs survive machine failures
Relevance quality Ranking changes must be observable and reviewable
Scale 1 B documents, 100 K queries/second, tens of TB of index data

Capacity Estimation

  1. Corpus size: 1 B documents at 10 KB each is ~10 TB of raw content.
  2. Index size: Inverted indexes plus stored fields, forward indexes, and ranking features often exceed source size, so plan for 20-50 TB.
  3. Crawl bandwidth: Refreshing 1 B documents every 30 days implies ~385 fetches/second on average, but practical systems need much higher burst capacity for hot domains.
  4. Query fan-out: At 100 K queries/second and 10 shards touched per query, the broker tier may issue ~1 M shard requests/second.
  5. Caching: If 20% of queries repeat and are cached, the system can avoid a large amount of ranking work.

Core Pipeline

  1. Crawler discovers pages.
  2. Fetcher downloads content.
  3. Parser extracts text, links, metadata, and canonical URLs.
  4. Deduper checks canonical URL and content hash.
  5. Indexer builds or updates inverted-index segments.
  6. Ranker combines lexical relevance with authority and freshness signals.
  7. Query serving returns the top results.

Ranking Signals

  • BM25 or TF-IDF for lexical relevance.
  • PageRank or link-based authority.
  • Freshness for recency-sensitive content.
  • Personalization where appropriate.
  • Click-through and dwell time as secondary behavioral signals.
  • Spam and quality classifiers to demote manipulated pages.

Query Serving

  • Use an inverted index for term lookup.
  • Keep hot postings lists in memory.
  • Precompute common query rewrites and snippets where helpful.
  • Example: for the query redis consistency, fetch postings for both terms, intersect or union candidate documents, score them, and return the top results.

Design Axes

  • Scalability: Crawling and indexing scale horizontally across workers and storage nodes.
  • Latency vs throughput: Query serving is latency-sensitive; crawl and index pipelines are throughput-sensitive.
  • Availability vs consistency: Search can tolerate slightly stale results if that keeps the query service available.
  • Reliability and fault tolerance: Distributed crawlers, replicated indexes, and retries prevent pipeline collapse.
  • Stateless vs stateful: Query workers are mostly stateless; indexes and ranking features are stateful.
  • Synchronous vs asynchronous: Indexing is asynchronous; query serving is synchronous.

Deduplication The act of detecting and ignoring repeated requests or messages so duplicates do not create duplicate side effects.
TTL (Time To Live) The time after which cached data, sessions, or records expire automatically.
Hot key A key that receives a disproportionate share of traffic and can overload one shard, cache node, or partition.
Cache stampede A burst of simultaneous cache misses for the same hot key that overwhelms the backing store.
Sharding Splitting data across multiple machines so storage and traffic scale horizontally.
Circuit breaker A protection pattern that stops sending requests to a failing dependency until it recovers.
SLI / SLO / SLA SLI is a measured service metric, SLO is the internal target for that metric, and SLA is the external promise with contractual consequences.
RTO / RPO RTO is how quickly service must be restored after failure, and RPO is how much data loss is acceptable.
ACID Atomicity, Consistency, Isolation, Durability — the four properties that guarantee correct transaction behavior in relational databases.
WAL (Write-Ahead Log) A durability mechanism where every change is written to an append-only log before being applied to the main data file. If the system crashes, the WAL is replayed to recover the last consistent state.
Saga pattern A way to implement distributed transactions across microservices by splitting work into a chain of local transactions, each of which publishes an event or triggers the next step. If a step fails, compensating transactions roll back prior steps.
mTLS (mutual TLS) An extension of TLS where both client and server authenticate each other with certificates. Used for service-to-service trust inside a service mesh, replacing password or token auth at the transport layer.
CDC (Change Data Capture) Streaming database row-level changes (inserts, updates, deletes) to downstream consumers as they happen. Tools include Debezium for Postgres/MySQL and DynamoDB Streams. Useful for cache invalidation, event sourcing, and replication.
PACELC An extension of CAP that also considers the latency–consistency trade-off when the network is healthy (no partition). Even with no partition, a system must choose between lower latency and stronger consistency.
Fan-out Delivering one event or message to many downstream consumers. Example: one ORDER_PLACED event fans out to inventory, billing, notification, and analytics consumers, each reading from the same Kafka topic independently.
Blue-green deployment A release strategy that runs two identical production environments. Traffic is switched from the old version (blue) to the new version (green) atomically, with near-instant rollback by switching back.
Canary deployment A release strategy that routes a small percentage of traffic (e.g., 1–5%) to the new version and gradually increases it as confidence grows. Allows catching production bugs before they affect all users.
Bounded context A Domain-Driven Design concept: the explicit boundary within which a particular domain model is defined and applies. The same word (e.g., "Customer") can mean different things in different bounded contexts.
Schema registry A service that stores and enforces versioned message schemas (usually Avro or Protobuf) for events on a message bus. Producers validate schemas before publishing; consumers validate on consumption, preventing deserialization errors from schema drift.
Service mesh Infrastructure that manages service-to-service communication — including mTLS, retries, circuit breaking, and load balancing — at the platform layer rather than in application code. Examples: Istio, Linkerd.
Dead-letter queue (DLQ) A separate queue or topic where messages are moved after failing more than N retry attempts. Prevents one malformed message from blocking an entire queue indefinitely.
Consumer group A Kafka concept: a named set of consumers that collectively read from a topic. Each partition is assigned to exactly one consumer in the group, enabling parallel processing. Multiple groups can read the same topic independently at different offsets.
Inverted index A data structure mapping each term to the list of documents containing it. The core of full-text search engines: given a query term, an O(1) lookup returns the posting list without scanning every document.
Posting list The list of document IDs (and optionally positions and scores) stored under each term in an inverted index. Query execution intersects posting lists for multi-term queries.
Idempotency key A client-supplied unique token attached to a request so that if the same request is retried, the server can detect and return the original response instead of processing it twice. Critical for payment APIs and message producers.
Two-phase commit (2PC) A distributed protocol for atomic cross-node writes: a coordinator first asks all participants to "prepare" (Phase 1), then—only if all agree—tells them to "commit" (Phase 2). If any participant says no or crashes, the coordinator aborts. Guarantees atomicity but blocks if the coordinator crashes after Phase 1.
Gossip protocol A peer-to-peer failure detection and state propagation mechanism where each node periodically exchanges its known-state with random neighbors. Used in Cassandra and Consul to propagate ring membership without a single coordinator.
TF-IDF Term Frequency–Inverse Document Frequency. A relevance score that rewards terms appearing frequently in a specific document (TF) but penalizes terms appearing in almost every document (IDF), which makes them poor discriminators. Used in search ranking.
PageRank A link-analysis algorithm that scores pages by the quantity and quality of pages linking to them. High PageRank means authoritative sites consider this page worth linking to. Used as an authority signal in web search ranking alongside content relevance.
